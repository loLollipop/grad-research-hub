# 研途 Hub：理工科研究生工作台优化计划

## 1. 调研结论

研途 Hub 不应该做成“什么都能配、什么都能管”的后台系统。理工科研究生和博士生最缺的是一个每天打开后能立刻回答三个问题的工作台：

1. 今天先推进什么？
2. 哪些文献、实验、结果或事务需要收口？
3. 这些材料怎么自然进入组会、周报和论文写作？

因此，产品重点应是连接已有工具和日常断点，而不是替代成熟工具。

## 2. 参考工具与取舍

### Zotero

Zotero 已经很好地解决文献库、集合、标签、条目同步和附件管理。Zotero Web API v3 提供 items、collections、tags、搜索、分页和同步版本等能力，生产代码也应指定 API 版本。

研途 Hub 的取舍：

- 做 Zotero 同步、阅读状态、阅读计划、阅读笔记。
- 不重新发明 PDF 管理器。
- 不要求用户大量手动录入文献，手动补录只作为兜底。

### eLabFTW / ELN

eLabFTW 的实验记录强调实验条目、模板、状态、标签、正文、步骤和关联资源。对个人研究生来说，最高频价值不是完整 LIMS，而是让每次实验留下目的、方法、观察、结论和下一步。

研途 Hub 的取舍：

- 做轻量实验日志、模板正文、状态、失败复盘、结果回填。
- 不做仪器预约、库存、签名、复杂权限。
- 不把实验页变成参数仓库。

### OSF

OSF 的项目 components 思路适合按研究阶段、材料、数据、分析、手稿组织项目。个人研究生更需要的是课题路线图和近期推进，而不是完整协作仓库。

研途 Hub 的取舍：

- 做项目、里程碑、任务、推进笔记。
- 不做多人权限、文件仓库、DOI/注册流程。
- 任务界面保持轻，优先显示下一步。

### Obsidian

Obsidian 的双链和 backlinks 说明研究笔记的关键价值是“看见关联”。研途 Hub 可以吸收这个能力，但不必复制整个知识库生态。

研途 Hub 的取舍：

- 做 Markdown、`[[双链]]`、反向链接、最近笔记和写作素材。
- 不急着做复杂图谱、插件系统或完整编辑器。

### 研究数据管理 / FAIR

研究生常见痛点是数据组织、结果文档化、复现状态、图表路径和后续写作素材散落。个人工具不应先做完整数据平台，而应先帮用户判断“这条结果能不能汇报、能不能写进论文”。

研途 Hub 的取舍：

- 做结果证据台、复现状态、图表/路径、写作素材。
- 不做数据仓库、样本库存、文件上传和复杂元数据表单。

## 3. 高需求功能优先级

1. 今日指挥台：聚合任务、事务、实验、文献和成果，给出行动顺序。
2. Zotero 阅读台：同步文献后变成阅读队列和阅读计划。
3. 实验日志台：轻量模板、状态、失败复盘、结果回填。
4. 课题推进台：项目、里程碑、下一步任务，减少项目管理负担。
5. 笔记沉淀台：Markdown、双链、反向链接、写作素材和任务拆解。
6. 结果证据台：复现状态、图表路径、可写入论文的判断。
7. 事务减负台：组会、材料、报销、截止提醒，低负担收口。
8. 设置连接中心：只把 AI Key、Zotero Key、访问密码这类常变配置放到网页端。

暂不做：

- 完整日历系统。
- PDF 托管和批注。
- 多人协作权限。
- 复杂甘特图。
- RAG/自动科研代理。
- 大量可配置字段。

## 4. UI 设计方向

目标不是“后台管理系统”，而是“安静但有生命力的科研工作台”。界面需要更舒适、更有辨识度，也要适合长时间使用。

### 视觉关键词

- 纸面感：背景像干净的实验记录纸，而不是冷硬后台。
- 低疲劳：主色降低饱和度，用钴蓝、青绿、墨色和少量暖砂。
- 行动优先：第一屏先给下一步，而不是统计和说明。
- 密而不乱：保留信息密度，但用层级、留白和分组减压。
- 中文一致：界面文案保持中文，技术名词如 Zotero、AI 可保留。

### 组件原则

- 导航代表研究流程，不是后台菜单。
- 当前页面选中态必须明确，离开后恢复。
- 高频新增使用按钮和弹窗；笔记编辑和实验正文保留大编辑空间。
- 卡片不堆大段配置，优先展示“下一步动作”。
- 空状态给用户下一步，不写产品说明书。

## 5. 本轮落地计划

### A. 全局视觉系统

- 调整 CSS token：纸面背景、柔和边框、低饱和主色、青绿/暖砂辅助色。
- 增加轻纹理和柔和层次，但不使用图片生成。
- 把深色行动栈从硬黑蓝改为更舒适的钴蓝到青绿渐变。
- 卡片阴影变轻，减少后台感。

### B. 应用壳

- 侧边栏更像“研究流程卡”，减少机械感。
- 顶部快速捕捉更像命令入口，适合快速记录任务、实验、文献、组会。
- 移动端导航补上 AI 和设置入口。
- 提升导航 hover/active 的一致性。

### C. 首页

- 首页继续保持“今日指挥台”。
- 强化 01/02/03 行动栈，让用户一眼知道先做什么。
- 减少统计卡片的压迫感。

### D. 核心页面持续方向

- 文献：Zotero-first，阅读计划优先。
- 实验：目的/观察/结论/下一步优先。
- 项目：下一步任务优先。
- 笔记：右侧编辑区要撑开，像工作室，不像弹窗。
- 成果：复现和论文素材优先。
- 事务：减负，不做行政系统。
- AI：草稿助手，不做自动代理。

## 6. 验收标准

1. 桌面端和移动端无明显横向溢出。
2. 导航选中态切换正常。
3. 首页首屏能回答“今天先做什么”。
4. 文献、实验、项目、笔记等页面不再像表单后台。
5. 设置页只暴露常改配置，不让部署参数干扰日常使用。
6. `npm run lint` 和 `npm run build` 通过。
7. GitHub 与服务器同步到同一提交。

## 8. 第二轮视觉收口计划

上一轮已经把全局颜色、侧边栏、导航和行动栈从后台感改成低疲劳研究工作台。第二轮继续做“细节减噪”，不新增功能，专门处理那些会让用户觉得粗糙或割裂的地方。

### 本轮优先事项

1. 加载态：把普通灰色骨架换成和工作台一致的纸面骨架，避免切页时出现突兀的后台占位感。
2. 弹窗：把新增/编辑弹窗改成轻量工作面板，弱化遮罩压迫感，表头、正文和关闭按钮更清晰。
3. 表单：增强标签可读性，减少“配置表单”的机器感。
4. 语义色：抽出 `research-eyebrow`、`soft-tile`、`soft-chip`、`warm-chip`、`skeleton-block` 等全局类，降低页面内硬编码色块。
5. 保持边界：仍然不新增数据库表、不新增配置项、不引入图片生成或复杂动效。

### 判断标准

- 新增、编辑、加载这三类高频状态要和主界面气质一致。
- 页面切换时不再像传统后台骨架屏。
- 表单仍然紧凑，但要更像人填写的工作记录，而不是机器参数面板。
- 所有动效必须轻，并遵守 `prefers-reduced-motion`。

## 9. 第三轮调研更新与落地计划

### 调研更新

本轮继续确认一个边界：研途 Hub 不应该复制 Zotero、eLabFTW、OSF 或 Obsidian，而应该连接它们在研究生日常中的断点。

- Zotero Web API v3 已覆盖 items、collections、tags、按集合读取条目和 API key 权限检查，文献页继续坚持“Zotero 同步优先，手动补录兜底”。
- eLabFTW 这类 ELN 的高频价值在实验条目、模板和状态，而不是让个人研究生维护完整 LIMS。
- OSF components 的价值是把大课题拆成可描述、可复现的部分；研途 Hub 只吸收“课题容器 + 阶段推进”，不做协作仓库。
- Obsidian 的内链和 backlinks 说明笔记核心价值是关联可见；研途 Hub 保留 `[[双链]]` 和反向链接，不做复杂知识图谱。
- 研究数据管理与 FAIR 思路提醒我们：结果要能被找到、复现、解释和复用；个人平台先回答“这条结果能否汇报/写作”，不要上来做重型数据平台。

### 本轮优先落地

1. 全局视觉继续降疲劳：减少深色大块压迫感，把行动栈从“重后台深色卡”改成更柔和的钴蓝-青绿研究指挥条。
2. 侧边栏从模块菜单继续改成研究闭环：强调“文献 -> 实验 -> 成果 -> 写作”，让用户知道每天打开后平台服务什么流程。
3. 导航选中态更明确：当前页使用稳定深色选中态，离开后自动恢复，避免点过的标签残留深色。
4. 顶部快速捕捉更像研究命令入口：减少说明感，保留自然短句示例，帮助忙碌用户不用找页面也能记录。
5. 不新增表、不新增配置、不引入新依赖；只优化每天高频触达的工作台体验。

### 验收标准

- 侧边栏和顶部不再像通用后台管理系统。
- 深色区域仍有重点，但不压眼、不大面积黑蓝。
- 当前导航页清晰可见，切换后旧页面不保留深色状态。
- `npm run lint` 和 `npm run build` 通过。
- GitHub 与服务器同步到同一提交。

## 10. 第四轮落地：实验日志台减后台感

### 为什么先改实验页

理工科研究生最常见的日常断点之一是：实验做了，但记录没有收口；失败了，但没有复盘；结果有了，但没有回填到实验记录和后续汇报材料。相比继续增加字段，实验页更应该帮用户快速判断“这条实验今天该做什么”。

### 本轮调整

1. 左侧“状态分布”不再作为统计图展示，改成“今日实验收口”：继续观察、失败复盘、结果回填三个行动入口。
2. 每条实验卡增加“下一步动作条”，根据状态和结果数量直接提示：继续观察、复盘失败、补结果、回填结果。
3. 保留原有模板、筛选、复盘笔记、复盘任务和结果回填能力，不新增数据库表。
4. 页面文案继续坚持轻量 ELN 边界：目的、方法、观察、结论和下一步优先，不做参数仓库或 LIMS。

### 验收标准

- 打开实验页后，不需要先理解统计分布，就能看到今天该收口哪类实验。
- 失败实验有明确复盘入口，完成实验有明确补结果/回填结果提示。
- UI 更像实验工作台，而不是后台状态报表。
- `npm run lint` 和 `npm run build` 通过。

## 11. 第五轮落地：课题推进台行动解释层

### 为什么改项目页

项目页已经有任务、里程碑、看板和推进笔记，但研究生真正打开它时，最需要的不是完整项目管理，而是快速判断：今天先推进哪几件事、为什么是它们、下一步是挂里程碑还是转成实验。完整 Jira/Notion 式项目管理会增加维护成本，因此本轮只加强行动解释，不加重字段。

### 本轮调整

1. 在左侧增加“推进雷达”，把今天/逾期、高优先级、可转实验这些开工信号放在一起。
2. 下一步队列和任务卡增加“行动理由”，解释为什么这条任务现在该处理。
3. 保留已有看板、批量更新、挂载里程碑、转实验、推进笔记能力，不新增数据表。
4. UI 继续强调“课题路线图 + 下一步动作”，避免变成通用后台项目管理。

### 验收标准

- 打开项目页后，用户能先看到今天最该处理的风险和高价值任务。
- 每条任务不仅有状态标签，还有面向人的行动理由。
- 任务仍能一键转实验、挂里程碑、生成推进笔记。
- `npm run lint` 和 `npm run build` 通过。

## 12. 第六轮落地：结果证据台行动解释层

### 为什么改成果页

成果页如果只显示数据集、指标和结果列表，很容易变成低频维护的数据仓库。理工科研究生更需要的是快速判断：这条结果是否可信、缺不缺复现实验、有没有图表路径、能不能进入组会或论文写作。因此本轮继续沿用“行动优先”的工作台方向，不新增复杂字段，只增强证据状态的可见性。

### 本轮调整

1. 左侧新增“证据雷达”，把待判断、复现实验、待补素材、可写入四类信号放在一起。
2. 下一步证据队列增加行动理由，说明为什么当前结果需要复现、补路径或转写作素材。
3. 每张结果卡增加轻量行动说明条，让用户不用展开详情就能判断下一步。
4. 保留现有结果记录、生成待补任务、生成写作素材和汇报清单能力，不新增数据表和配置项。

### 验收标准

- 打开成果页后，用户能先看到哪些结果缺证据、哪些结果已经能汇报或写作。
- 每条关键结果不仅有复现状态，还有面向人的下一步解释。
- 页面仍然保持轻量结果登记，而不是重型数据管理系统。
- `npm run lint` 和 `npm run build` 通过。
