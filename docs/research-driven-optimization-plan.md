# 研途 Hub 调研驱动优化计划

## 调研摘要

研途 Hub 的目标不是替代 Zotero、Obsidian、eLabFTW、OSF 或通用项目管理软件，而是做一个理工科研究生每天愿意打开的轻量工作台。调研后得到的核心判断是：研究生最需要的不是更多字段，而是更少的决策、更顺的切换、更清楚的下一步，以及把文献、实验、结果、写作和轻事务之间的断点接起来。

## 参考工具与可借鉴能力

- Zotero：Web API v3 支持按库、集合、标签、搜索等方式读取文献条目，并建议生产代码指定 API 版本；研途 Hub 应继续把 Zotero 当作文献源头，而不是要求用户重复录入文献。参考：<https://www.zotero.org/support/dev/web_api/v3/basics>
- eLabFTW：实验条目是 ELN 核心，模板可节省重复实验创建时间，实验记录包含标题、分类、状态、标签、正文、步骤、链接和附件等；研途 Hub 只吸收“模板 + 状态 + 正文”的个人轻量部分，不做完整 ELN。参考：<https://doc.elabftw.net/docs/usage/user-guide/experiments/>、<https://doc.elabftw.net/docs/usage/user-guide/templates/>
- OSF：项目可用 components 按研究阶段、材料类型、数据、分析、手稿等拆分，并用 wiki 记录过程；研途 Hub 应借鉴“项目容器 + 阶段化推进”，但不做协作者权限、文件仓库和 DOI 流程。参考：<https://help.osf.io/article/622-utilize-components-for-detailed-project-management>
- Obsidian：内部链接和 backlinks 能把笔记变成知识网络；研途 Hub 保留 `[[双链]]` 识别、最近笔记和写作工作区即可，不急着做完整知识图谱。参考：<https://obsidian.md/help/links>、<https://obsidian.md/help/plugins/backlinks>
- 研究数据管理研究：多项 RDM 研究指出研究生在数据组织、文档化、保存、共享、数据管理计划等方面存在能力缺口；研途 Hub 的“结果证据台”应强调可复盘、可写入论文、可追溯，而不是变成复杂数据平台。参考：<https://pmc.ncbi.nlm.nih.gov/articles/PMC9949669/>、<https://journals.library.ualberta.ca/istl/index.php/istl/article/download/2678/2709>

## 高需求功能优先级

1. 今日工作台：打开后先知道“下一步做什么”，将任务、实验、文献、事务截止合并排序。
2. Zotero 阅读台：同步优先，手动补录只是兜底；强化阅读状态、标签、集合筛选、批量状态更新。
3. 实验日志：让用户快速写清目的、方法、观察、结论、下一步；模板选择要比字段堆叠更重要。
4. 课题路线图：项目页面重点保留阶段、里程碑、任务和最近推进，不展示过量元信息。
5. 笔记工作室：保留沉浸式编辑，不把写笔记塞进弹窗；强化最近笔记、双链、预览和引用关系。
6. 结果证据台：围绕“这条结果能不能支撑论文”组织指标、数据集、复现状态和素材路径。
7. 轻事务时间线：组会、材料、报销、截止提醒按时间线处理，避免散落在微信、备忘录和表格里。
8. 设置中心：AI Key、Base URL、Zotero Key、访问密码放网页端改；数据库、端口、加密密钥留给部署配置。

## UI 设计原则

- 第一屏显示行动建议，不显示说明书。
- 低饱和纸面背景 + 靛蓝主色 + 青绿/暖砂辅助色，避免长时间使用视觉疲劳。
- 导航是“研究流程”，不是后台菜单；当前页面必须明确高亮。
- 列表、卡片和表单保持紧凑，但留足阅读空间。
- 表单默认收起到按钮或弹窗，笔记编辑器和实验正文除外。
- 首页用“下一步 -> 文献输入 -> 实验推进 -> 结果证据”的科研节奏条解释产品，而不是用营销文案解释功能。

## 当前落地轮次

- 全局视觉 token 调整为低疲劳纸面/靛蓝/青绿体系。
- 侧边栏增加轻量研究流程提示，导航选中态更清楚。
- 顶部快速捕捉改成更像命令入口，适合记录想法、待办和组会提醒。
- Dashboard 增加“科研节奏条”，将下一步、文献、实验、证据串成每日工作流。
- 保持数据库结构不变，避免为了 UI 改造引入迁移风险。

## 下一轮建议

1. 文献台：增加集合快捷筛选、批量改阅读状态、最近同步版本/错误细节。
2. 实验日志：模板选择后动态预填正文，失败实验一键生成复盘任务。
3. 设置中心：增加 Zotero 连接测试、AI 连接测试和更清晰的 ready/not ready 反馈。
4. 结果证据台：增加“可写入论文”证据清单和按项目/实验过滤。
5. 视觉验证：用桌面与移动端截图审查页面拥挤、空白、文字溢出和选中态一致性。

## 第二十九轮落地：结果到写作素材

- 调研依据：研究数据管理相关研究反复强调研究生在数据组织、文档化、保存和复用上存在压力；对个人科研工作台来说，最直接的落点不是做完整数据平台，而是让关键结果能快速变成周报、组会和论文素材。
- 功能边界：不新增表、不新增配置、不替代实验记录或论文写作软件，只把成果页的一条结果整理成一篇可继续编辑的写作笔记。
- 交互调整：每张结果卡片和论文素材卡片增加“写作素材”按钮，点击后进入笔记工作室，而不是在成果页继续堆表单。
- 去重策略：笔记正文写入隐藏标记，同一条结果重复点击会打开已有素材笔记，避免生成重复草稿。
- 素材结构：自动带入项目、实验、数据集、核心指标、复现状态、图表路径、关联文献、图表检查清单和后续动作。
- 产品价值：把“有结果了但不知道怎么整理成可讲内容”的高频痛点收口成一步，贴合理工科研究生组会前、周报前和论文初稿前的真实工作流。

## 第三十轮落地：今日开工清单

- 调研依据：理工科研究生的日常压力集中在“今天先做什么、哪些实验/结果别忘收口、组会前材料怎么串起来”。比起完整日历或复杂 GTD，首页应优先降低每日开工前的选择成本。
- 功能边界：不做新日历、不做重复提醒系统、不新增数据库表；只把当前已有任务、事务、实验、结果和文献整理成当天可编辑的 Markdown 笔记。
- 交互调整：首页主操作增加“生成今日计划”，已生成时直接显示“打开今日计划”；组会入口降级成辅助操作，避免第一屏动作过多。
- 去重策略：今日计划使用日期隐藏标记，同一天重复点击会打开已有笔记，避免每天生成多份相似清单。
- 素材结构：自动列出今天最重要三件事、顺手收口项、进行中实验、最近结果、待读/读中文献和晚上收口检查。
- 产品价值：把“开电脑后先逛一遍项目/实验/文献/事务”的碎片动作收成一步，让用户先进入工作状态，再按需跳转到具体模块。

## 第三十一轮落地：首页 cockpit 化

- 调研依据：数据密集型工作台应优先支持快速扫描和比较；研究生每天打开平台时，需要先获得“当前焦点、风险信号、文献队列和证据素材”，而不是先阅读产品说明。
- UI 方向：把首页第一屏从介绍型 hero 改为 cockpit 布局，左侧直接显示今天最值得处理的事项，右侧显示实时信号卡。
- 信息层级：今日计划、收口提醒、文献队列和证据素材统一放在右侧小面板，减少首页横向通知条堆叠。
- 视觉策略：继续使用低饱和纸面背景、靛蓝/青绿信号色和轻透明面板；不使用强渐变、深色大块或装饰性图形，降低长时间使用疲劳。
- 交互策略：首页主按钮保留“打开/生成今日计划”，次级动作才是今日任务、实验记录和组会草稿，避免第一屏像后台菜单集合。

## 第三十二轮落地：文献阅读计划

- 调研依据：Zotero 适合作为文献条目、集合和标签的源头，但研究生日常常卡在“当前这批文献怎么读、读完沉淀到哪里”。因此研途 Hub 不重复管理 PDF，而是把当前筛选队列变成可执行阅读计划。
- 功能边界：不新增表、不抓 PDF、不做 RAG；只把当前文献列表前 12 篇整理成一篇 Markdown 笔记。
- 交互调整：文献页 hero 和列表工具条增加“整理/生成阅读计划”，空列表时给出轻提示；生成后直接进入笔记工作室继续编辑。
- 去重策略：阅读计划使用日期和文献 ID 集合生成隐藏标记，同一天同一批文献重复点击会打开已有计划。
- 素材结构：自动列出共同问题、每篇文献需要读出的问题/方法/关键结果/可复用点，以及读完后的项目任务和实验动作。
- 产品价值：服务组会前、开题前和读文献日的真实流程，减少“同步了一堆文献但不知道先读哪几篇”的负担。

## 第三十三轮落地：课题推进笔记

- 调研依据：OSF 的 components 思路适合按研究阶段组织材料，但个人研究生更需要轻量的“本周/今天推进什么”。课题路线图不应变成重型项目管理，而应帮用户把任务、里程碑和实验动作串起来。
- 功能边界：不新增表、不做甘特图、不做多人协作权限；只把项目页当前任务队列前 12 个整理成一篇 Markdown 推进笔记。
- 交互调整：项目页 hero 和任务工具条增加“生成推进笔记”，空任务时给出提示；生成后直接进入笔记工作室继续编辑。
- 去重策略：推进笔记使用日期和任务 ID 集合生成隐藏标记，同一天同一批任务重复点击会打开已有笔记。
- 素材结构：按项目分组列出任务、里程碑、状态、优先级、截止和备注，并附上推进后回填清单。
- 产品价值：服务周计划、导师沟通和组会复盘，让任务队列从“看板卡片”变成可以直接讨论和执行的研究计划。

## 第三十四轮落地：笔记反向链接面板

- 调研依据：Obsidian 的内部链接和 backlinks 说明了研究笔记最有价值的不是复杂编辑器，而是能看见“这条想法连向哪里、又被哪里引用”。研途 Hub 只吸收这个高频价值点，不复制完整知识图谱。
- 功能边界：不新增数据库表、不做图谱可视化、不加入新的配置项；继续用 Markdown 正文中的 `[[双链标题]]` 自动解析关系。
- 交互调整：笔记页“双链”页签拆成当前引用、未创建链接、已匹配笔记和反向链接四块，用户不用额外维护关系，也能发现可继续补齐的主题。
- 匹配策略：基于全量笔记标题做大小写无关匹配，而不是只看当前搜索/分类结果；同名标题优先指向最近更新的一条。
- 产品价值：服务文献摘录、实验复盘和写作素材之间的轻量连接，让研究生在写笔记时自然沉淀上下文，而不是被迫进入复杂知识库维护流程。

## 第三十五轮落地：首页研究指挥台与低疲劳视觉

- 调研依据：Zotero Web API 已覆盖文献库、条目、集合和标签，eLabFTW 类 ELN 强调实验模板与状态，OSF 的 components 适合按研究阶段组织项目，Obsidian 强调双链/反向链接，RDM 研究持续强调数据组织、文档化、保存与复用。研途 Hub 的高需求位置仍然是把这些工具之间的日常断点接起来，而不是替代它们。
- 本轮计划：优先改首页和全局视觉，因为这是研究生每天打开平台的第一触点；第一屏必须回答“今天先做什么、有什么要收口、文献/实验/成果现在卡在哪里”。
- 功能边界：不新增数据库表、不新增配置项、不新增复杂模块；继续复用任务、事务、文献、实验、结果和笔记的现有数据，生成行动信号。
- UI 调整：全局背景从明显网格纹理改成柔和层次纸面；侧栏改成“文献 -> 实验 -> 成果”的科研流程提示；首页首屏从传统 cockpit 文案改为“今日指挥台”，新增 01/02/03 今日行动栈。
- 产品价值：让忙碌的理工科研究生打开后先看到行动顺序，而不是先读统计卡片或配置说明；同时用更冷静的钴蓝/青绿视觉降低长时间使用疲劳。

## 第三十六轮落地：Zotero 阅读栈

- 调研依据：Zotero 的强项是文献库、集合、标签、条目同步和附件管理；研究生真正卡住的日常问题通常不是“再建一个文献库”，而是“今天读哪几篇、读到什么状态、读完怎么沉淀成笔记”。
- 本轮计划：把文献页从后台式列表继续改成“阅读台”，第一屏显示今日阅读栈，侧栏增加待读、读中、已读和全部文献的快捷视图。
- 功能边界：不新增数据库表、不托管 PDF、不做 RAG，不替代 Zotero 的 collections/tags；继续复用现有 `Paper.readStatus`、`category` 和阅读笔记生成能力。
- UI 调整：文献页 hero 统一到研究指挥台视觉；阅读栈用 01/02/03 的深色行动面板；文献卡片增加“开始阅读 / 继续阅读 / 回顾笔记”的下一步提示。
- 产品价值：减少“同步了一堆文献但不知道先读哪篇”的负担，让文献流自然进入阅读笔记、组会和课题推进。

## 第三十七轮落地：实验日志栈

- 调研依据：eLabFTW 这类 ELN 的核心不是堆参数，而是让实验有模板、状态、标签和可追踪上下文；实验记录模板也反复强调目的、方法、观察、结论和下一步。
- 本轮计划：把实验页从实验列表继续改成“实验日志台”，第一屏显示今日实验栈，优先呈现进行中和失败实验；侧栏增加全部实验、继续观察、已完成和失败复盘快捷视图。
- 功能边界：不新增数据库表、不做 LIMS、样品库存或仪器预约；继续复用现有实验状态、模板、复盘笔记、失败复盘任务和结果回填能力。
- UI 调整：实验页 hero 统一到研究指挥台视觉；实验卡片增加“继续观察 / 补结果 / 沉淀结论 / 复盘失败”的下一步提示。
- 产品价值：解决“实验做完了但记录散、失败没复盘、结果没回填”的高频痛点，让实验流自然进入结果证据和组会材料。

## 第三十八轮落地：课题推进栈

- 调研依据：OSF 的 project/components/wiki/files 思路适合组织研究阶段和材料，但个人研究生更需要轻量的“今天先推进什么”；完整项目管理、文件归档和多人协作不应挤进首版日常工作台。
- 本轮计划：把项目页从任务看板继续改成“课题推进台”，第一屏显示今日推进栈，优先呈现当前关键词/项目下未完成的高价值任务；侧栏增加全部任务、今天/逾期、高优先级、进行中和本周收口快捷视图。
- 功能边界：不新增数据库表、不做甘特图、权限、文件仓库或多人协作；继续复用现有项目、里程碑、任务、推进笔记和任务转实验能力。
- UI 调整：项目页 hero 统一到研究指挥台视觉；任务卡片增加“开始推进 / 继续推进 / 优先处理 / 先补逾期 / 补里程碑 / 已收口”的下一步提示。
- 产品价值：减少“课题、里程碑、任务都在，但打开后还是不知道先做什么”的摩擦，让项目流自然进入实验动作、推进笔记和组会沟通。

## 第三十九轮落地：结果证据栈

- 调研依据：研究数据管理和 FAIR 原则强调结果要可查找、可访问、可互操作、可复用；但个人研究生日常最痛的不是维护完整元数据，而是“哪些结果还没复现、哪些已经能汇报、哪些缺图表或路径”。
- 本轮计划：把成果页从数据/结果列表继续改成“结果证据台”，第一屏显示今日证据栈，优先呈现待复现、复现中、缺图表路径或尚未标记为写作素材的结果。
- 功能边界：不新增数据库表、不做数据仓库、样本库存、文件上传或复杂 FAIR 表单；继续复用现有结果、实验、数据集、写作素材笔记和证据任务能力。
- UI 调整：成果页 hero 统一到研究指挥台视觉；侧栏增加全部结果、待判断、待复现、复现中、已复现和可写入快捷视图；结果卡片增加“先判定复现 / 补复现实验 / 继续复现 / 补图表路径 / 标成写作素材 / 已成证据”的下一步提示。
- 产品价值：减少“跑出了指标但不知道是否能讲、能不能写、缺什么证据”的摩擦，让实验结果自然进入组会清单、写作素材和项目推进任务。

## 第四十轮落地：笔记沉淀栈

- 调研依据：Obsidian 的内部链接和 backlinks 说明，研究笔记的高频价值不在复杂编辑器，而在“写下材料、看见关联、把清单拆回任务”。理工科研究生更需要阅读摘录、实验复盘、结果证据和组会草稿之间顺手流转。
- 本轮计划：把笔记页从写作工作室继续改成“沉淀台”，第一屏显示今日沉淀栈，优先呈现含未拆待办、缺失双链、最近更新和写作素材类笔记。
- 功能边界：不新增数据库表、不引入 Tiptap/MDX Editor、不做知识图谱可视化；继续复用现有 Markdown、`[[双链]]`、反向链接面板和待办清单拆任务能力。
- UI 调整：笔记页 hero 统一到研究指挥台视觉；最近笔记卡片增加“拆任务 / 补双链 / 继续打磨 / 回顾关系 / 继续写”的下一步提示；右侧大编辑器继续保留，不改成弹窗。
- 产品价值：减少“文献、实验、结果都生成了笔记，但不知道该继续整理哪篇”的摩擦，让笔记自然回到项目任务、组会复盘和论文写作。

## 第四十一轮落地：事务减负栈

- 调研依据：研究生日常行政负担通常不是复杂流程，而是组会、材料、报销和截止提醒在微信、邮件、表格和脑子里来回散落，打断科研专注。个人工作台应把这些小事收进低负担队列，而不是变成完整行政系统。
- 本轮计划：把事务页从统计时间线继续改成“减负台”，第一屏显示今日减负栈，优先呈现逾期、今天截止、进行中和组会相关事项。
- 功能边界：不新增数据库表、不做完整日历、不做审批流、不做多人协作；继续复用现有事务类型、状态、截止、组会草稿生成和轻量筛选能力。
- UI 调整：事务页 hero 统一到研究指挥台视觉；侧栏增加全部事务、今天/逾期、组会、材料、报销、待处理和已完成快捷视图；事务卡增加“先补逾期 / 今天处理 / 继续推进 / 准备组会 / 补报销材料 / 整理材料 / 已收口”的下一步提示。
- 产品价值：减少“明明是小事却总惦记着”的心理负担，让用户快速处理事务，然后回到文献、实验、结果和写作。

## 第四十二轮落地：AI 草稿助手栈

- 调研依据：研究生常用 AI 辅助学术写作、文献综述、想法整理、研究计划和文本润色，但也担心隐私、事实错误、原创性和过度依赖。个人科研工作台应把 AI 放在“整理草稿、用户核对”的位置，而不是做自动科研代理。
- 本轮计划：把 AI 页从配置说明和试验台继续改成“草稿助手台”，第一屏显示今日助手栈，围绕组会、实验复盘、阅读卡片和写作润色四个高频场景。
- 功能边界：不新增数据库表、不自动读取用户隐私材料、不做长链代理；继续复用现有 AI 设置、接口骨架和提示词工作台。
- UI 调整：AI 页 hero 统一到研究指挥台视觉；侧栏改为场景入口和连接状态；页面明确提示先脱敏，AI 输出只当草稿，事实、引用和结论必须人工核对。
- 产品价值：减少“我要问 AI 什么、怎么组织提示词”的成本，让 AI 回到组会准备、实验复盘、阅读提炼和写作打磨这些高频但低风险的日常环节。

## 第四十三轮落地：设置连接中心

- 调研依据：开源自托管工具的设置页最容易劝退用户，尤其是研究生用户只想修改常换的 Key、同步源和访问密码，不想理解所有部署变量。设置页应把日常连接和一次性部署信息分层。
- 本轮计划：把设置页从传统配置中心改成“连接中心”，第一屏显示今日连接栈，优先呈现 AI、Zotero 和访问保护的可用性。
- 功能边界：不新增数据库表、不把数据库 URL、端口、加密密钥做成网页端可编辑项；继续复用现有设置、测试连接、导出和高级部署说明。
- UI 调整：设置页 hero 统一到研究指挥台视觉；左侧显示当前状态和备份入口；右侧保留 AI、Zotero、访问密码三个常改表单；高级部署信息继续折叠。
- 产品价值：减少部署后第一次打开平台的配置焦虑，让用户先确认“能同步、能登录、能导出”，再进入文献、实验、项目和笔记工作流。

## 第四十四轮落地：组会/导师周报准备

- 调研依据：理工科研究生的高频沟通不是完整项目汇报系统，而是每周把“做了什么、证据在哪、卡在哪里、下周怎么做”快速整理给导师或组会。个人工作台应减少会前翻任务、翻实验、翻结果和翻文献的时间。
- 本轮计划：把原有“组会草稿”升级为“组会/周报准备”，继续复用任务、事务、实验、结果、文献和笔记数据，一键生成可编辑 Markdown。
- 功能边界：不新增数据库表、不做审批式周报、不做复杂模板配置；生成后进入笔记页继续编辑，用户可以按导师习惯删减内容。
- UI 调整：首页和事务页按钮统一为“生成周报草稿”；事务页第一屏改成“组会/周报减负台”，弱化后台感，强化导师沟通场景。
- 草稿结构：先写一句话进展、需要导师确认和下周最小计划，再列可展示证据、实验/数据更新、文献输入、阻塞与事务提醒，最后保留会后回填清单。
- 产品价值：让研究生在组会前不用临时拼材料，也不用维护一个新系统；打开平台即可把本周散落记录变成可汇报、可复盘、可继续拆任务的草稿。

## 第四十五轮落地：写作素材包

- 调研依据：文献管理、电子实验记录和研究数据管理工具分别解决“存在哪里”和“怎么记录”，但研究生写论文、周报或开题材料时最耗时的是把结果、文献、笔记和组会反馈重新拼成可写文本。
- 本轮计划：在笔记页增加“生成写作素材包”，把已标记或带图表路径的结果、读中/已读文献、最近阅读/实验/结果/组会/写作笔记汇成一篇 Markdown。
- 功能边界：不新增数据库表、不做自动论文写作、不引入复杂编辑器或模板配置；素材包当天复用，重复点击直接打开已有笔记。
- UI 调整：笔记页第一屏保留大编辑器，左侧新增轻量提示条和按钮，强调“生成后仍在右侧编辑”，避免把写作变成新的后台模块。
- 草稿结构：先选择今天要写哪一段，再列可写入结果证据、文献输入、最近可用笔记和初稿骨架。
- 产品价值：减少“明明平台里有结果和阅读笔记，但写作时还要到处翻”的摩擦，让成果、文献和组会自然流向论文/周报初稿。
