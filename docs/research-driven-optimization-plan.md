# 研途 Hub 调研驱动优化计划

## 调研摘要

研途 Hub 的目标不是替代 Zotero、Obsidian、eLabFTW、OSF 或通用项目管理软件，而是做一个理工科研究生每天愿意打开的轻量工作台。调研后得到的核心判断是：研究生最需要的不是更多字段，而是更少的决策、更顺的切换、更清楚的下一步，以及把文献、实验、结果、写作和轻事务之间的断点接起来。

## 参考工具与可借鉴能力

- Zotero：Web API v3 支持按库、集合、标签、搜索等方式读取文献条目，并建议生产代码指定 API 版本；研途 Hub 应继续把 Zotero 当作文献源头，而不是要求用户重复录入文献。参考：<https://www.zotero.org/support/dev/web_api/v3/basics>
- eLabFTW：实验条目是 ELN 核心，模板可节省重复实验创建时间，实验记录包含标题、分类、状态、标签、正文、步骤、链接和附件等；研途 Hub 只吸收“模板 + 状态 + 正文”的个人轻量部分，不做完整 ELN。参考：<https://doc.elabftw.net/docs/usage/user-guide/experiments/>、<https://doc.elabftw.net/docs/usage/user-guide/templates/>
- OSF：项目可用 components 按研究阶段、材料类型、数据、分析、手稿等拆分，并用 wiki 记录过程；研途 Hub 应借鉴“项目容器 + 阶段化推进”，但不做协作者权限、文件仓库和 DOI 流程。参考：<https://help.osf.io/article/622-utilize-components-for-detailed-project-management>
- Obsidian：内部链接和 backlinks 能把笔记变成知识网络；研途 Hub 保留 `[[双链]]` 识别、最近笔记和写作工作区即可，不急着做完整知识图谱。参考：<https://obsidian.md/help/links>、<https://obsidian.md/help/plugins/backlinks>
- 研究数据管理研究：多项 RDM 研究指出研究生在数据组织、文档化、保存、共享、数据管理计划等方面存在能力缺口；研途 Hub 的“结果证据台”应强调可复盘、可写入论文、可追溯，而不是变成复杂数据平台。参考：<https://pmc.ncbi.nlm.nih.gov/articles/PMC9949669/>、<https://journals.library.ualberta.ca/istl/index.php/istl/article/download/2678/2709>

## 高需求功能优先级

1. 今日工作台：打开后先知道“下一步做什么”，将任务、实验、文献、事务截止合并排序。
2. Zotero 阅读台：同步优先，手动补录只是兜底；强化阅读状态、标签、集合筛选、批量状态更新。
3. 实验日志：让用户快速写清目的、方法、观察、结论、下一步；模板选择要比字段堆叠更重要。
4. 课题路线图：项目页面重点保留阶段、里程碑、任务和最近推进，不展示过量元信息。
5. 笔记工作室：保留沉浸式编辑，不把写笔记塞进弹窗；强化最近笔记、双链、预览和引用关系。
6. 结果证据台：围绕“这条结果能不能支撑论文”组织指标、数据集、复现状态和素材路径。
7. 轻事务时间线：组会、材料、报销、截止提醒按时间线处理，避免散落在微信、备忘录和表格里。
8. 设置中心：AI Key、Base URL、Zotero Key、访问密码放网页端改；数据库、端口、加密密钥留给部署配置。

## UI 设计原则

- 第一屏显示行动建议，不显示说明书。
- 低饱和暖纸背景 + 石板蓝灰主色 + 鼠尾草绿/暖砂辅助色，避免长时间使用视觉疲劳。
- 导航是“研究流程”，不是后台菜单；当前页面必须明确高亮。
- 列表、卡片和表单保持紧凑，但留足阅读空间。
- 表单默认收起到按钮或弹窗，笔记编辑器和实验正文除外。
- 首页用“下一步 -> 文献输入 -> 实验推进 -> 结果证据”的科研节奏条解释产品，而不是用营销文案解释功能。

## 当前落地轮次

- 全局视觉 token 调整为低疲劳暖纸/石板蓝灰/鼠尾草绿体系。
- 侧边栏增加轻量研究流程提示，导航选中态更清楚。
- 顶部快速捕捉改成更像命令入口，适合记录想法、待办和组会提醒。
- Dashboard 增加“科研节奏条”，将下一步、文献、实验、证据串成每日工作流。
- 保持数据库结构不变，避免为了 UI 改造引入迁移风险。

## 下一轮建议

1. 文献台：增加集合快捷筛选、批量改阅读状态、最近同步版本/错误细节。
2. 实验日志：模板选择后动态预填正文，失败实验一键生成复盘任务。
3. 设置中心：增加 Zotero 连接测试、AI 连接测试和更清晰的 ready/not ready 反馈。
4. 结果证据台：增加“可写入论文”证据清单和按项目/实验过滤。
5. 视觉验证：用桌面与移动端截图审查页面拥挤、空白、文字溢出和选中态一致性。

## 第二十九轮落地：结果到写作素材

- 调研依据：研究数据管理相关研究反复强调研究生在数据组织、文档化、保存和复用上存在压力；对个人科研工作台来说，最直接的落点不是做完整数据平台，而是让关键结果能快速变成周报、组会和论文素材。
- 功能边界：不新增表、不新增配置、不替代实验记录或论文写作软件，只把成果页的一条结果整理成一篇可继续编辑的写作笔记。
- 交互调整：每张结果卡片和论文素材卡片增加“写作素材”按钮，点击后进入笔记工作室，而不是在成果页继续堆表单。
- 去重策略：笔记正文写入隐藏标记，同一条结果重复点击会打开已有素材笔记，避免生成重复草稿。
- 素材结构：自动带入项目、实验、数据集、核心指标、复现状态、图表路径、关联文献、图表检查清单和后续动作。
- 产品价值：把“有结果了但不知道怎么整理成可讲内容”的高频痛点收口成一步，贴合理工科研究生组会前、周报前和论文初稿前的真实工作流。

## 第三十轮落地：今日开工清单

- 调研依据：理工科研究生的日常压力集中在“今天先做什么、哪些实验/结果别忘收口、组会前材料怎么串起来”。比起完整日历或复杂 GTD，首页应优先降低每日开工前的选择成本。
- 功能边界：不做新日历、不做重复提醒系统、不新增数据库表；只把当前已有任务、事务、实验、结果和文献整理成当天可编辑的 Markdown 笔记。
- 交互调整：首页主操作增加“生成今日计划”，已生成时直接显示“打开今日计划”；组会入口降级成辅助操作，避免第一屏动作过多。
- 去重策略：今日计划使用日期隐藏标记，同一天重复点击会打开已有笔记，避免每天生成多份相似清单。
- 素材结构：自动列出今天最重要三件事、顺手收口项、进行中实验、最近结果、待读/读中文献和晚上收口检查。
- 产品价值：把“开电脑后先逛一遍项目/实验/文献/事务”的碎片动作收成一步，让用户先进入工作状态，再按需跳转到具体模块。

## 第三十一轮落地：首页 cockpit 化

- 调研依据：数据密集型工作台应优先支持快速扫描和比较；研究生每天打开平台时，需要先获得“当前焦点、风险信号、文献队列和证据素材”，而不是先阅读产品说明。
- UI 方向：把首页第一屏从介绍型 hero 改为 cockpit 布局，左侧直接显示今天最值得处理的事项，右侧显示实时信号卡。
- 信息层级：今日计划、收口提醒、文献队列和证据素材统一放在右侧小面板，减少首页横向通知条堆叠。
- 视觉策略：继续使用低饱和纸面背景、靛蓝/青绿信号色和轻透明面板；不使用强渐变、深色大块或装饰性图形，降低长时间使用疲劳。
- 交互策略：首页主按钮保留“打开/生成今日计划”，次级动作才是今日任务、实验记录和组会草稿，避免第一屏像后台菜单集合。

## 第三十二轮落地：文献阅读计划

- 调研依据：Zotero 适合作为文献条目、集合和标签的源头，但研究生日常常卡在“当前这批文献怎么读、读完沉淀到哪里”。因此研途 Hub 不重复管理 PDF，而是把当前筛选队列变成可执行阅读计划。
- 功能边界：不新增表、不抓 PDF、不做 RAG；只把当前文献列表前 12 篇整理成一篇 Markdown 笔记。
- 交互调整：文献页 hero 和列表工具条增加“整理/生成阅读计划”，空列表时给出轻提示；生成后直接进入笔记工作室继续编辑。
- 去重策略：阅读计划使用日期和文献 ID 集合生成隐藏标记，同一天同一批文献重复点击会打开已有计划。
- 素材结构：自动列出共同问题、每篇文献需要读出的问题/方法/关键结果/可复用点，以及读完后的项目任务和实验动作。
- 产品价值：服务组会前、开题前和读文献日的真实流程，减少“同步了一堆文献但不知道先读哪几篇”的负担。

## 第三十三轮落地：课题推进笔记

- 调研依据：OSF 的 components 思路适合按研究阶段组织材料，但个人研究生更需要轻量的“本周/今天推进什么”。课题路线图不应变成重型项目管理，而应帮用户把任务、里程碑和实验动作串起来。
- 功能边界：不新增表、不做甘特图、不做多人协作权限；只把项目页当前任务队列前 12 个整理成一篇 Markdown 推进笔记。
- 交互调整：项目页 hero 和任务工具条增加“生成推进笔记”，空任务时给出提示；生成后直接进入笔记工作室继续编辑。
- 去重策略：推进笔记使用日期和任务 ID 集合生成隐藏标记，同一天同一批任务重复点击会打开已有笔记。
- 素材结构：按项目分组列出任务、里程碑、状态、优先级、截止和备注，并附上推进后回填清单。
- 产品价值：服务周计划、导师沟通和组会复盘，让任务队列从“看板卡片”变成可以直接讨论和执行的研究计划。

## 第三十四轮落地：笔记反向链接面板

- 调研依据：Obsidian 的内部链接和 backlinks 说明了研究笔记最有价值的不是复杂编辑器，而是能看见“这条想法连向哪里、又被哪里引用”。研途 Hub 只吸收这个高频价值点，不复制完整知识图谱。
- 功能边界：不新增数据库表、不做图谱可视化、不加入新的配置项；继续用 Markdown 正文中的 `[[双链标题]]` 自动解析关系。
- 交互调整：笔记页“双链”页签拆成当前引用、未创建链接、已匹配笔记和反向链接四块，用户不用额外维护关系，也能发现可继续补齐的主题。
- 匹配策略：基于全量笔记标题做大小写无关匹配，而不是只看当前搜索/分类结果；同名标题优先指向最近更新的一条。
- 产品价值：服务文献摘录、实验复盘和写作素材之间的轻量连接，让研究生在写笔记时自然沉淀上下文，而不是被迫进入复杂知识库维护流程。

## 第三十五轮落地：首页研究指挥台与低疲劳视觉

- 调研依据：Zotero Web API 已覆盖文献库、条目、集合和标签，eLabFTW 类 ELN 强调实验模板与状态，OSF 的 components 适合按研究阶段组织项目，Obsidian 强调双链/反向链接，RDM 研究持续强调数据组织、文档化、保存与复用。研途 Hub 的高需求位置仍然是把这些工具之间的日常断点接起来，而不是替代它们。
- 本轮计划：优先改首页和全局视觉，因为这是研究生每天打开平台的第一触点；第一屏必须回答“今天先做什么、有什么要收口、文献/实验/成果现在卡在哪里”。
- 功能边界：不新增数据库表、不新增配置项、不新增复杂模块；继续复用任务、事务、文献、实验、结果和笔记的现有数据，生成行动信号。
- UI 调整：全局背景从明显网格纹理改成柔和层次纸面；侧栏改成“文献 -> 实验 -> 成果”的科研流程提示；首页首屏从传统 cockpit 文案改为“今日指挥台”，新增 01/02/03 今日行动栈。
- 产品价值：让忙碌的理工科研究生打开后先看到行动顺序，而不是先读统计卡片或配置说明；同时用更冷静的钴蓝/青绿视觉降低长时间使用疲劳。

## 第三十六轮落地：Zotero 阅读栈

- 调研依据：Zotero 的强项是文献库、集合、标签、条目同步和附件管理；研究生真正卡住的日常问题通常不是“再建一个文献库”，而是“今天读哪几篇、读到什么状态、读完怎么沉淀成笔记”。
- 本轮计划：把文献页从后台式列表继续改成“阅读台”，第一屏显示今日阅读栈，侧栏增加待读、读中、已读和全部文献的快捷视图。
- 功能边界：不新增数据库表、不托管 PDF、不做 RAG，不替代 Zotero 的 collections/tags；继续复用现有 `Paper.readStatus`、`category` 和阅读笔记生成能力。
- UI 调整：文献页 hero 统一到研究指挥台视觉；阅读栈用 01/02/03 的深色行动面板；文献卡片增加“开始阅读 / 继续阅读 / 回顾笔记”的下一步提示。
- 产品价值：减少“同步了一堆文献但不知道先读哪篇”的负担，让文献流自然进入阅读笔记、组会和课题推进。

## 第三十七轮落地：实验日志栈

- 调研依据：eLabFTW 这类 ELN 的核心不是堆参数，而是让实验有模板、状态、标签和可追踪上下文；实验记录模板也反复强调目的、方法、观察、结论和下一步。
- 本轮计划：把实验页从实验列表继续改成“实验日志台”，第一屏显示今日实验栈，优先呈现进行中和失败实验；侧栏增加全部实验、继续观察、已完成和失败复盘快捷视图。
- 功能边界：不新增数据库表、不做 LIMS、样品库存或仪器预约；继续复用现有实验状态、模板、复盘笔记、失败复盘任务和结果回填能力。
- UI 调整：实验页 hero 统一到研究指挥台视觉；实验卡片增加“继续观察 / 补结果 / 沉淀结论 / 复盘失败”的下一步提示。
- 产品价值：解决“实验做完了但记录散、失败没复盘、结果没回填”的高频痛点，让实验流自然进入结果证据和组会材料。

## 第三十八轮落地：课题推进栈

- 调研依据：OSF 的 project/components/wiki/files 思路适合组织研究阶段和材料，但个人研究生更需要轻量的“今天先推进什么”；完整项目管理、文件归档和多人协作不应挤进首版日常工作台。
- 本轮计划：把项目页从任务看板继续改成“课题推进台”，第一屏显示今日推进栈，优先呈现当前关键词/项目下未完成的高价值任务；侧栏增加全部任务、今天/逾期、高优先级、进行中和本周收口快捷视图。
- 功能边界：不新增数据库表、不做甘特图、权限、文件仓库或多人协作；继续复用现有项目、里程碑、任务、推进笔记和任务转实验能力。
- UI 调整：项目页 hero 统一到研究指挥台视觉；任务卡片增加“开始推进 / 继续推进 / 优先处理 / 先补逾期 / 补里程碑 / 已收口”的下一步提示。
- 产品价值：减少“课题、里程碑、任务都在，但打开后还是不知道先做什么”的摩擦，让项目流自然进入实验动作、推进笔记和组会沟通。

## 第三十九轮落地：结果证据栈

- 调研依据：研究数据管理和 FAIR 原则强调结果要可查找、可访问、可互操作、可复用；但个人研究生日常最痛的不是维护完整元数据，而是“哪些结果还没复现、哪些已经能汇报、哪些缺图表或路径”。
- 本轮计划：把成果页从数据/结果列表继续改成“结果证据台”，第一屏显示今日证据栈，优先呈现待复现、复现中、缺图表路径或尚未标记为写作素材的结果。
- 功能边界：不新增数据库表、不做数据仓库、样本库存、文件上传或复杂 FAIR 表单；继续复用现有结果、实验、数据集、写作素材笔记和证据任务能力。
- UI 调整：成果页 hero 统一到研究指挥台视觉；侧栏增加全部结果、待判断、待复现、复现中、已复现和可写入快捷视图；结果卡片增加“先判定复现 / 补复现实验 / 继续复现 / 补图表路径 / 标成写作素材 / 已成证据”的下一步提示。
- 产品价值：减少“跑出了指标但不知道是否能讲、能不能写、缺什么证据”的摩擦，让实验结果自然进入组会清单、写作素材和项目推进任务。

## 第四十轮落地：笔记沉淀栈

- 调研依据：Obsidian 的内部链接和 backlinks 说明，研究笔记的高频价值不在复杂编辑器，而在“写下材料、看见关联、把清单拆回任务”。理工科研究生更需要阅读摘录、实验复盘、结果证据和组会草稿之间顺手流转。
- 本轮计划：把笔记页从写作工作室继续改成“沉淀台”，第一屏显示今日沉淀栈，优先呈现含未拆待办、缺失双链、最近更新和写作素材类笔记。
- 功能边界：不新增数据库表、不引入 Tiptap/MDX Editor、不做知识图谱可视化；继续复用现有 Markdown、`[[双链]]`、反向链接面板和待办清单拆任务能力。
- UI 调整：笔记页 hero 统一到研究指挥台视觉；最近笔记卡片增加“拆任务 / 补双链 / 继续打磨 / 回顾关系 / 继续写”的下一步提示；右侧大编辑器继续保留，不改成弹窗。
- 产品价值：减少“文献、实验、结果都生成了笔记，但不知道该继续整理哪篇”的摩擦，让笔记自然回到项目任务、组会复盘和论文写作。

## 第四十一轮落地：事务减负栈

- 调研依据：研究生日常行政负担通常不是复杂流程，而是组会、材料、报销和截止提醒在微信、邮件、表格和脑子里来回散落，打断科研专注。个人工作台应把这些小事收进低负担队列，而不是变成完整行政系统。
- 本轮计划：把事务页从统计时间线继续改成“减负台”，第一屏显示今日减负栈，优先呈现逾期、今天截止、进行中和组会相关事项。
- 功能边界：不新增数据库表、不做完整日历、不做审批流、不做多人协作；继续复用现有事务类型、状态、截止、组会草稿生成和轻量筛选能力。
- UI 调整：事务页 hero 统一到研究指挥台视觉；侧栏增加全部事务、今天/逾期、组会、材料、报销、待处理和已完成快捷视图；事务卡增加“先补逾期 / 今天处理 / 继续推进 / 准备组会 / 补报销材料 / 整理材料 / 已收口”的下一步提示。
- 产品价值：减少“明明是小事却总惦记着”的心理负担，让用户快速处理事务，然后回到文献、实验、结果和写作。

## 第四十二轮落地：AI 草稿助手栈

- 调研依据：研究生常用 AI 辅助学术写作、文献综述、想法整理、研究计划和文本润色，但也担心隐私、事实错误、原创性和过度依赖。个人科研工作台应把 AI 放在“整理草稿、用户核对”的位置，而不是做自动科研代理。
- 本轮计划：把 AI 页从配置说明和试验台继续改成“草稿助手台”，第一屏显示今日助手栈，围绕组会、实验复盘、阅读卡片和写作润色四个高频场景。
- 功能边界：不新增数据库表、不自动读取用户隐私材料、不做长链代理；继续复用现有 AI 设置、接口骨架和提示词工作台。
- UI 调整：AI 页 hero 统一到研究指挥台视觉；侧栏改为场景入口和连接状态；页面明确提示先脱敏，AI 输出只当草稿，事实、引用和结论必须人工核对。
- 产品价值：减少“我要问 AI 什么、怎么组织提示词”的成本，让 AI 回到组会准备、实验复盘、阅读提炼和写作打磨这些高频但低风险的日常环节。

## 第四十三轮落地：设置连接中心

- 调研依据：开源自托管工具的设置页最容易劝退用户，尤其是研究生用户只想修改常换的 Key、同步源和访问密码，不想理解所有部署变量。设置页应把日常连接和一次性部署信息分层。
- 本轮计划：把设置页从传统配置中心改成“连接中心”，第一屏显示今日连接栈，优先呈现 AI、Zotero 和访问保护的可用性。
- 功能边界：不新增数据库表、不把数据库 URL、端口、加密密钥做成网页端可编辑项；继续复用现有设置、测试连接、导出和高级部署说明。
- UI 调整：设置页 hero 统一到研究指挥台视觉；左侧显示当前状态和备份入口；右侧保留 AI、Zotero、访问密码三个常改表单；高级部署信息继续折叠。
- 产品价值：减少部署后第一次打开平台的配置焦虑，让用户先确认“能同步、能登录、能导出”，再进入文献、实验、项目和笔记工作流。

## 第四十四轮落地：组会/导师周报准备

- 调研依据：理工科研究生的高频沟通不是完整项目汇报系统，而是每周把“做了什么、证据在哪、卡在哪里、下周怎么做”快速整理给导师或组会。个人工作台应减少会前翻任务、翻实验、翻结果和翻文献的时间。
- 本轮计划：把原有“组会草稿”升级为“组会/周报准备”，继续复用任务、事务、实验、结果、文献和笔记数据，一键生成可编辑 Markdown。
- 功能边界：不新增数据库表、不做审批式周报、不做复杂模板配置；生成后进入笔记页继续编辑，用户可以按导师习惯删减内容。
- UI 调整：首页和事务页按钮统一为“生成周报草稿”；事务页第一屏改成“组会/周报减负台”，弱化后台感，强化导师沟通场景。
- 草稿结构：先写一句话进展、需要导师确认和下周最小计划，再列可展示证据、实验/数据更新、文献输入、阻塞与事务提醒，最后保留会后回填清单。
- 产品价值：让研究生在组会前不用临时拼材料，也不用维护一个新系统；打开平台即可把本周散落记录变成可汇报、可复盘、可继续拆任务的草稿。

## 第四十五轮落地：写作素材包

- 调研依据：文献管理、电子实验记录和研究数据管理工具分别解决“存在哪里”和“怎么记录”，但研究生写论文、周报或开题材料时最耗时的是把结果、文献、笔记和组会反馈重新拼成可写文本。
- 本轮计划：在笔记页增加“生成写作素材包”，把已标记或带图表路径的结果、读中/已读文献、最近阅读/实验/结果/组会/写作笔记汇成一篇 Markdown。
- 功能边界：不新增数据库表、不做自动论文写作、不引入复杂编辑器或模板配置；素材包当天复用，重复点击直接打开已有笔记。
- UI 调整：笔记页第一屏保留大编辑器，左侧新增轻量提示条和按钮，强调“生成后仍在右侧编辑”，避免把写作变成新的后台模块。
- 草稿结构：先选择今天要写哪一段，再列可写入结果证据、文献输入、最近可用笔记和初稿骨架。
- 产品价值：减少“明明平台里有结果和阅读笔记，但写作时还要到处翻”的摩擦，让成果、文献和组会自然流向论文/周报初稿。

## 第四十六轮落地：自然语言快速捕捉

- 调研依据：忙碌的研究生最容易丢失的不是完整记录，而是临时想到的下一步、实验观察、待读文献、周报句子和行政提醒；捕捉入口必须比打开页面和填表更快。
- 本轮计划：增强顶部快速捕捉，不再要求用户必须输入 `任务：` 这类冒号前缀，也支持 `任务 整理结果`、`实验 复现第3组`、`写作 周报开头`、`结果 图表待补` 等自然短句。
- 功能边界：不做聊天机器人、不做复杂 NLP、不新增配置项；只做稳定的轻前缀和短词分流，识别不到时仍进入收件箱笔记。
- UI 调整：侧栏和顶部输入改成“随手写，自动分流”的命令条提示，给出真实科研场景示例，减少用户记忆成本。
- 分流策略：任务进入独立待办，实验生成目的/方法/观察/结论模板，文献进入待读，事务按组会/材料/报销/截止推断类型，写作/周报/结果进入对应笔记分类。
- 产品价值：让平台更像每天打开就能用的科研工作台，而不是一个需要先理解模块和表单的后台系统。

## 第四十七轮落地：Zotero 分页同步与排查提示

- 调研依据：Zotero Web API v3 支持对条目接口使用 `limit`、`start` 等分页参数；研究生的 Zotero 库经常超过单页返回数量，如果同步只取第一页，会让用户误以为平台漏文献。
- 本轮计划：把 Zotero 同步改成按设置的同步数量自动分页读取，最多仍受设置中心的同步数量限制，避免一次请求过大。
- 功能边界：不新增数据库表、不做复杂增量版本管理、不托管 PDF；继续以 Zotero 为文献源头，研途 Hub 只维护阅读状态和笔记沉淀。
- UI 调整：文献页“阅读节奏”增加同步范围说明，显示当前是同步顶层文献还是限定 Collection Key，以及单次最多同步多少条。
- 错误定位：把 400、401/403、404、429 等常见 Zotero API 错误转成更像人能处理的中文提示，减少用户来回猜 Key、Library ID 和 Collection Key。
- 产品价值：让“配置 Zotero -> 点击同步 -> 看见阅读队列”更可靠，降低开源自托管用户第一次接入文献库时的挫败感。

## 第四十八轮落地：Zotero 同步摘要

- 调研依据：Zotero Web API v3 的多对象读取会在响应头返回 `Total-Results`，单次响应最多 100 条；仅显示“同步成功”不足以解释为什么大型文献库没有一次全部出现。
- 本轮计划：同步后把读取到的 Zotero 条目数、实际导入文献数、当前范围总数、同步范围和“是否还有更多”一起带回文献页。
- 功能边界：不新增数据库表、不保存 Zotero 同步游标、不暴露 API Key；继续用设置中心的同步数量作为用户可理解的拉取上限。
- UI 调整：成功提示从单句“已同步 N 条”升级为同步摘要；如果仍有更多条目，会提示调高同步数量或用 Collection Key 分批同步。
- 产品价值：减少第一次接入 Zotero 时的“不知道是不是漏了”的不信任感，让用户更放心地把 Zotero 作为文献源头。

## 第四十九轮落地：真实 AI 草稿调用

- 调研依据：研究生高频使用 AI 的位置是组会提纲、实验复盘、阅读提炼和写作润色，但风险在于事实幻觉、引用错误和隐私泄露。
- 本轮计划：把 AI 页从占位试验台升级为真实草稿助手，读取设置中心的 Provider、Base URL、模型名和 Key，调用 OpenAI-compatible Chat Completions 或 Anthropic Messages。
- 功能边界：不做自动科研代理、不自动读取用户私密数据、不做 RAG；只处理用户主动粘贴的脱敏材料，并明确输出需要人工核对。
- UI 调整：按钮从“试运行”改为“生成草稿”，结果区区分真实草稿和配置提示，未配置 Key 时返回可理解的下一步。
- 产品价值：让设置中心里经常变动的 AI Key 真的服务日常科研草稿，而不是停留在配置展示。

## 第五十轮落地：AI 草稿回流笔记

- 调研依据：AI 生成内容如果只停留在聊天框里，研究生仍然要复制、整理、核对和归档；更高频的真实动作是把可用草稿沉淀到笔记，再继续补证据。
- 本轮计划：AI 草稿生成后提供“保存到笔记”，直接创建写作分类笔记，保留原始提示、草稿内容和核对清单。
- 功能边界：不新增表、不自动覆盖旧笔记、不把 AI 输出当最终结论；保存后的笔记默认带“待核对”标签。
- UI 调整：AI 页文案强调草稿可以回到笔记工作室；结果卡片只在真实模型输出后显示保存按钮。
- 产品价值：把 AI 从孤立助手变成研究闭环的一环，让组会、实验复盘、阅读卡片和写作润色自然进入可编辑、可双链、可拆任务的笔记系统。

## 第五十一轮落地：低疲劳长工作区

- 调研依据：Zotero、eLabFTW、OSF、Obsidian 和研究数据管理资料都说明，研究生真正高频使用的是连续工作流：同步文献、写实验、收结果、补笔记、准备组会。页面如果像后台表单一样只显示短卡片和大片空白，会增加切换成本，也会让长时间使用更疲劳。
- 本轮计划：先不新增功能，把事务页和 AI 页的右侧工作区拉伸到底部，并把 AI 工作台从单个文本框升级为“输入材料 / 可核对草稿”的双栏工作区。
- 功能边界：不新增数据库表、不新增配置项、不自动读取隐私数据；AI 仍然只处理用户主动粘贴的脱敏材料，输出仍需人工核对。
- UI 调整：全局纸面网格进一步降低对比度，双栏页面使用统一 `workbench-dual-grid` 和 `workbench-column`；事务列表、AI 输入区和 AI 草稿结果区都能撑满可用高度，减少页面底部悬空。
- 交互调整：AI 场景入口改为可扫读模板按钮；输入区显示字数和隐私提示；结果区没有内容时显示空状态，有结果时支持直接保存到笔记。
- 产品价值：让长时间处理事务、组会草稿和 AI 写作辅助时更像一个研究工作室，而不是零散后台页面；对忙碌研究生而言，少一点视觉噪音和空白，就是少一点额外认知负担。

## 第五十二轮落地：10 分钟开箱清单

- 调研依据：研究生/博士生的工具迁移成本很高，第一次打开平台时如果先看到多个模块和配置入口，很容易把它当成又一个需要维护的系统。高需求工作台应该先降低启动摩擦，让用户只完成文献源、当前课题和第一条记录这三个动作。
- 本轮计划：首页空库状态增加“生成上手清单”主操作，点击后生成一篇可编辑 Markdown 笔记，而不是要求用户先读 README 或在多个页面之间来回跳。
- 功能边界：不新增数据库表、不新增配置项、不创建示例假数据；清单只是可勾选笔记，重复点击会打开已有清单。
- UI 调整：空库第一屏的首要动作从“先填设置”调整为“生成上手清单”，10 分钟顺序的第一步也变成可点击生成清单。
- 清单结构：今天原则、连接 Zotero、建真实课题、留下第一条研究记录、晚上收口和暂时不要做的事项。
- 产品价值：让新用户从“我该先配置什么”变成“我只需要照着清单完成三件小事”，更符合忙碌理工科研究生的实际上手节奏。

## 第五十三轮落地：成果证据缺口

- 调研依据：研究数据管理资料强调组织、文档化、保存和复用，但个人研究生日常最常见的问题是“这个结果到底能不能讲、能不能写、还缺什么”。成果页不应继续扩展成数据仓库，而应优先暴露证据缺口。
- 本轮计划：在成果页列表上方新增三张证据缺口卡：先判定、补复现、补素材。
- 功能边界：不新增数据库表、不新增配置、不改变结果表单；继续复用已有复现状态、图表路径和写作素材标记。
- UI 调整：证据缺口卡放在筛选器之前，让用户先看到最需要处理的结果类型，再决定是否过滤列表。
- 产品价值：减少“数据页没啥用”的感受，让成果页直接回答组会/周报前最关键的问题：哪些结果还不能作为证据。

## 第五十四轮落地：实验记录纸面化

- 调研依据：eLabFTW 等 ELN 的高频价值在实验条目、模板、状态和可追踪正文，而不是让个人研究生维护完整 LIMS、仓库链接或运行参数。用户真正需要的是快速写清目的、观察、结论和下一步。
- 本轮计划：重构新建/编辑实验弹窗里的表单视觉，把它从字段表单改成“实验记录纸”。
- 功能边界：不新增字段、不改数据库、不展示 Git、仓库、运行号等低频技术字段；继续保留项目、状态、模板、关联论文、标签和 Markdown 正文。
- UI 调整：标题区变成实验卡片，项目/状态/模板变成紧凑控制组，正文区变成大面积纸面编辑器，底部只保留保存动作和后续流转提示。
- 产品价值：让实验页更贴近理工科研究生真实记录习惯，减少“这是给机器填参数”的感觉。

## 第五十五轮落地：临时文献卡片

- 调研依据：Zotero 已经覆盖文献库、集合、标签、附件和引用信息，研途 Hub 不应让用户重复维护完整文献库。手动补录只服务导师临时发来的网页、预印本、课题相关材料等少量临时条目。
- 本轮计划：把“手动补录文献”弹窗从完整文献字段表单改成轻量临时文献卡片。
- 功能边界：不新增字段、不替代 Zotero、不做 DOI/arXiv 自动抓取；继续复用标题、作者、年份、来源、DOI、外链、标签、摘要和阅读笔记。
- UI 调整：标题和“为什么要读”成为主入口，作者/年份/来源/DOI 变成可补充信息，底部明确提醒正式文献建议回 Zotero 管理。
- 产品价值：减少“同步 Zotero 之外还要手填一堆文献信息”的负担，让文献页更专注阅读状态和读后沉淀。

## 第五十六轮落地：下一步行动卡

- 调研依据：OSF 的 components/wiki 适合把研究项目拆成阶段和材料，但个人研究生最常见的摩擦不是“项目结构不够完整”，而是“打开项目页后下一步到底做什么”。任务入口如果像后台字段表，会让用户把本该快速捕捉的行动写成项目说明。
- 设计依据：本轮使用 `ui-ux-pro-max` 生成并持久化研途 Hub 设计系统，方向为 data-dense、slate-cobalt、compact panels and precise separators；重点是快速扫描、低疲劳、对比清楚，而不是营销式大卡片。
- 本轮计划：把项目页的新建/编辑任务弹窗从普通表单改成“下一步行动卡”，让标题成为第一入口，里程碑、优先级、状态和截止集中到紧凑控制区。
- 功能边界：不新增数据库表、不做甘特图、不做完整项目管理、不新增配置项；继续复用任务、里程碑、任务转实验和推进笔记能力。
- UI 调整：任务标题区加入行动说明和图标；归属路线、截止、优先级、状态使用更像按钮组/下拉控制的布局；备注改名为“必要上下文”，提醒只写验收标准、数据位置、相关文献或导师确认问题。
- 产品价值：减少“为了加一条任务还要维护很多信息”的阻力，让任务更贴近理工科研究生每天真正需要的下一步行动，并自然流向实验、推进笔记和今日队列。

## 第五十七轮落地：事务减负卡

- 调研依据：理工科研究生的行政事务通常不是复杂审批，而是组会提醒、材料入口、报销单号和硬截止散落在聊天记录、邮件和脑子里。事务页应帮助用户释放注意力，而不是变成新的后台台账。
- 本轮计划：把新增/编辑事务弹窗从普通字段表单改成“事务减负卡”，让用户先写要从脑子里移走的小事，再补类型、状态、截止、渠道和必要上下文。
- 功能边界：不新增表、不做完整日历、不做审批流程、不做复杂通知；继续复用事务类型、状态、截止、周报草稿生成和现有筛选。
- UI 调整：标题区增加轻量说明和 Inbox 图标；类型/状态/截止集中到紧凑控制组；地点/渠道和标签降为辅助信息；备注改成“处理所需上下文”，引导记录入口、单号、确认人和缺项。
- 产品价值：减少“为了记一件小事还要填表”的抗拒感，让组会、材料、报销和截止能快速进入减负栈，并在周报草稿里自然出现。

## 第五十八轮落地：结果证据卡

- 调研依据：FAIR 和研究数据管理资料强调可追溯、可复用和文档化，但个人研究生最需要的不是完整数据仓库，而是快速判断“这条结果能不能支撑组会、周报或论文结论”。结果录入口应围绕证据质量，而不是堆日志字段。
- 本轮计划：把新增/编辑结果弹窗从线性字段表单改成“结果证据卡”，先写可讲清楚的结果，再补实验/数据集/复现状态、1-3 个核心指标、图表路径和下一步。
- 功能边界：不新增数据库表、不上传文件、不做 FAIR 元数据全量表单、不替代实验日志；继续复用现有结果、数据集、实验关联、写作素材和证据任务能力。
- UI 调整：标题区强调关键结论；实验、数据集、复现状态组成紧凑证据来源控制区；指标区改成证据指标卡；素材路径和“可写入”判断并排显示；备注改为“一句话结论与下一步”。
- 产品价值：减少“成果页像数据后台、录入后不知道有什么用”的感觉，让每条结果天然服务组会汇报、复现实验和论文素材沉淀。

## 第五十九轮落地：课题目标卡与阶段验收卡

- 调研依据：OSF 的 components 思路说明大型研究需要按阶段、材料和产出拆分，但个人研究生不需要完整项目仓库和权限系统。上手时最重要的是先建立真实课题，再拆出一个能验收的阶段。
- 本轮计划：把项目页的新增/编辑项目、里程碑弹窗从普通字段表单改成“课题目标卡”和“阶段验收卡”，让用户先写研究主线、近期交付物和阶段验收标准。
- 功能边界：不新增字段、不做甘特图、不做多人协作、不做完整 OSF 替代品；继续复用项目、里程碑、任务队列、任务转实验和推进笔记。
- UI 调整：项目标题区强调“建立真实课题”，描述区改为研究问题/近期交付物/最小下一步；里程碑标题区强调“可验收阶段”，状态和截止使用紧凑控制组；底部提示保存后只需补少量下一步行动。
- 产品价值：降低第一次上手时“我要维护一个复杂项目系统”的压力，让课题路线图更像研究计划，而不是后台台账。

## 第六十轮落地：数据来源卡

- 调研依据：研究数据管理资料强调数据组织、版本、位置和复用，但个人研究生不需要在日常工作台里维护完整数据资产目录。成果页的数据集登记只应服务“这条结果以后能不能追溯和复现”。
- 本轮计划：把新增/编辑数据集弹窗从资产登记表改成“数据来源卡”，先写会影响结果复现的数据来源，再补来源、版本、路径、外部链接和复现说明。
- 功能边界：不上传文件、不做数据仓库、不做权限和共享流程、不新增配置；继续复用现有数据集、结果关联、图表路径和结果证据卡。
- UI 调整：主标题区强调关键数据版本；来源/版本组成紧凑控制组；位置和外部链接只记录可找回路径；说明区改为“复现说明”，引导写用途、处理状态和依赖结果。
- 产品价值：让数据集功能从“看起来没啥用的表”变成结果复现的轻量锚点，减少写论文和组会前找不到数据来源的摩擦。

## 第六十一轮落地：连接卡式设置中心

- 调研依据：开源自托管项目第一次使用最容易卡在 Key、Library ID、Base URL 和密码这些连接项；研究生用户想知道“现在能不能同步/能不能生成草稿”，而不是理解一长串部署变量。
- 本轮计划：把设置中心里 AI 和 Zotero 两个高频表单改成“连接卡”，先显示当前是否可用和下一步，再分层展示关键输入、可选范围和测试动作。
- 功能边界：不新增设置项、不把数据库 URL/端口/加密密钥做成网页端可编辑、不暴露密钥；继续复用现有保存、测试、加密和反馈逻辑。
- UI 调整：AI 卡先说明 Key、模型和 Base URL 的用途，测试按钮贴近 API Key；Zotero 卡先说明平台只同步条目/集合/标签，Library ID、库类型、API Key 放主区，Collection Key 和同步数量放范围区。
- 产品价值：降低部署后第一次打开设置页的焦虑，让用户按“接 AI -> 接 Zotero -> 测试连接”的顺序完成最少步骤，而不是面对配置清单。

## 第六十二轮落地：AI 安全草稿卡

- 调研依据：研究生使用生成式 AI 的高频场景集中在组会提纲、实验复盘、阅读提炼和写作润色，但真实风险是隐私泄露、事实幻觉、引用错误和过度依赖。AI 页应强调“用户主动粘贴材料 + 输出人工核对”，而不是像通用聊天框。
- 本轮计划：重写 AI 场景模板，让每个模板都明确只基于粘贴材料、不补充未提供事实、材料不足时列待补问题，并加入固定输出结构。
- 功能边界：不新增模型配置、不自动读取平台数据、不做 RAG、不生成引用；继续复用现有 AI 调用、草稿保存到笔记和设置中心连接。
- UI 调整：场景按钮显示简短用途说明；输入区增加脱敏、只基于粘贴材料、人工核对三枚提示芯片；页面文案强调“主动粘贴的脱敏材料”。
- 产品价值：让 AI 功能贴合理工科研究生真实高频草稿需求，同时降低误用风险，使输出自然回流到笔记工作室继续编辑和核对。

## 第六十三轮落地：低疲劳纸面视觉

- 调研依据：研究生会长时间停留在同一工作台里处理文献、实验、结果、笔记和事务，界面如果网格太重、阴影太硬或色块太跳，会让平台更像后台系统，也更容易造成视觉疲劳。
- 本轮计划：先不新增功能，只调整全局纸面底色、卡片层级、侧栏和行动区视觉，让每天打开的第一层体验更轻、更稳、更适合长时间扫描。
- 功能边界：不新增数据库表、不新增配置项、不改变导航和业务流程；继续复用现有工作台组件与页面结构。
- UI 调整：全局背景改为更冷静的浅纸面层次，降低网格对比度和大面积阴影；卡片、双栏工作区、侧栏品牌标识和 cockpit 区域统一为更柔和的钴蓝/青绿色信号体系。
- 产品价值：减少“界面太吵、看久了累”的摩擦，让研途 Hub 更像持续使用的研究工作室，而不是一次性填表后台。

## 第六十四轮落地：今日研究流时间带

- 调研依据：Zotero、ELN、OSF、Obsidian 和研究数据管理资料共同说明，研究生真正高频的不是维护更多字段，而是在一天里反复完成“定下一步 -> 读文献/做实验 -> 收口观察 -> 准备组会/写作素材”的连续动作。
- 本轮计划：首页保留今日指挥台，但把原来的四个静态节奏卡升级为“今日研究流”时间带，按开工前、深度工作、收口前和组会前组织已有信号。
- 功能边界：不新增数据库表、不新增配置、不改变路由；继续复用任务、文献、实验、结果和周报草稿能力。
- UI 调整：时间带使用低饱和纸面背景、细连接线和紧凑行动节点，让页面更像可照着走的研究日程，而不是一组后台统计卡。
- 产品价值：让忙碌的理工科研究生打开首页后能更自然地进入当天节奏，减少在多个页面之间判断“现在该干什么”的认知成本。

## 第六十五轮落地：导师沟通准备度

- 调研依据：博士/研究生与导师沟通前最常见的准备结构是近期进展、关键证据、下周计划和需要导师判断的问题。组会/周报功能如果只堆材料，会让用户仍然需要从长文档里二次整理。
- 本轮计划：在事务页增加“导师沟通准备度”，用现有周报草稿、结果证据、高优先级任务、组会事务和待读文献推导 3 步准备状态。
- 功能边界：不新增表、不新增配置、不做完整日历或会议系统；继续复用事务页、周报草稿、笔记和现有统计。
- UI 调整：准备度以 01/02/03 的紧凑卡片呈现，突出“先有草稿 -> 再看证据 -> 最后列问题”；周报草稿模板前置“本次沟通先看这里”和“5 分钟汇报版本”。
- 产品价值：让组会/导师沟通从“材料堆”变成可直接编辑的沟通稿，减少会前临时翻任务、实验、结果和文献的时间。

## 第六十六轮落地：文献转实验桥

- 调研依据：Zotero 已经解决文献收集，ELN 负责实验记录；研究生日常真正卡住的地方，是读到一个方法、对照或指标后没有立刻落成可执行实验。
- 本轮计划：在文献卡片增加“转实验”动作，一键生成轻量复现实验草稿，并自动关联来源文献。
- 功能边界：不新增数据库表、不做 DOI/arXiv 抓取、不替代 Zotero、不做自动论文理解；只把用户已选文献变成可编辑实验草稿。
- UI 调整：阅读雷达和文献卡片文案改为强调“读后方法/对照/指标 -> 实验草稿”，按钮保留在单篇卡片动作区，避免干扰批量阅读状态。
- 产品价值：让文献输入自然流向实验验证，减少“读完了但没有下一步”的断点，使平台更贴合理工科研究生从论文到实验的真实工作流。

## 第六十七轮落地：实验生成结果证据

- 调研依据：研究数据管理和 ELN 实践都强调实验结果需要可追溯、可复现、可解释；研究生日常最容易断在“实验写了观察，但没有登记成能汇报的结果证据”。
- 本轮计划：在实验卡片的结果区增加“生成结果证据”，把实验观察一键转成成果页的结果证据草稿，并自动关联来源实验。
- 功能边界：不新增表、不上传文件、不做数据仓库、不替代 MLflow/DVC；只复用现有 Result 表和成果页证据卡。
- UI 调整：实验无结果时给出主动作“生成结果证据”；已有结果时保留“回填正文”，并提供“再记一条证据”作为低干扰动作。
- 产品价值：让实验日志自然流向组会/周报/论文证据，减少用户在实验页和成果页之间重复选择关联实验的摩擦。

## 第六十八轮落地：文献综述矩阵

- 调研依据：研究生做开题、组会和论文 related work 时，高频需求是横向比较一批文献的问题、方法、数据、指标和可复用点，而不是继续维护 PDF 或重复录入文献信息。
- 本轮计划：在文献页增加“生成综述矩阵”，把当前筛选列表前 12 篇文献整理成可编辑 Markdown 对比表。
- 功能边界：不新增表、不做自动综述、不生成未经用户确认的事实；矩阵只填入已有标题、作者、来源、阅读状态和现有摘要/笔记，其余栏位留给用户人工补齐。
- UI 调整：入口放在文献列表工具条，与阅读计划并列；空列表时给出轻提示，不打断 Zotero-first 工作流。
- 产品价值：让文献阅读自然流向综述、开题和论文写作，减少“读了很多但横向关系没沉淀”的摩擦。

## 第六十九轮落地：科研闭环快速捕捉

- 调研依据：研究生日常很多信息不是从完整表单开始的，而是突然想到一条结果、发现一个数据版本、临时收到一个任务。快速入口如果只能进笔记，会让后续仍然需要二次搬运。
- 本轮计划：扩展顶部快速捕捉，让“结果/成果/证据”直接生成成果页的结果证据草稿，“数据/数据集”直接生成数据来源草稿。
- 功能边界：不新增表、不新增配置、不替代成果页和数据来源卡；只把已有 Result、Dataset 能力接入同一个随手捕捉入口。
- UI 调整：顶部输入框示例加入结果和数据写法；成果页增加快速捕捉成功提示，明确刚刚创建的是结果证据还是数据来源。
- 产品价值：把“任务 -> 文献 -> 实验 -> 结果 -> 数据来源”的科研闭环收进一个入口，减少忙碌时在多个页面之间找按钮和搬运内容的摩擦。

## 第七十轮落地：首页研究时间线

- 调研依据：Zotero 的价值在文献集合、标签和条目同步；eLabFTW 强调实验条目和模板；OSF components 强调按研究阶段记录方法、结果和决策；Obsidian 的双链/反链说明笔记价值在上下文连接。研途 Hub 不重复这些工具，而是把它们之间最近发生的研究动作串起来。
- 本轮计划：首页“研究时间线”从普通最近列表升级成周报/组会前可回看的研究轨迹，统计最近文献、实验、结果和笔记数量，并按时间带展示类型、上下文、更新时间和下一步动作。
- 功能边界：不新增表、不新增配置、不做完整审计日志；继续复用现有 Paper、Experiment、Result 和 Note 的更新时间。
- UI 调整：时间线改成低疲劳纸面时间带，使用细连接线、类型标签和紧凑动作胶囊，减少大卡片堆叠造成的视觉疲劳。
- 产品价值：让研究生不用临时翻多个页面，就能在首页回看最近研究脉络，并自然进入周报、组会和写作整理。

## 第七十一轮落地：笔记素材筛选

- 调研依据：Obsidian 的双链/反链价值在于让笔记上下文可追踪，但研究生写周报或论文前最需要的不是维护完整知识图谱，而是快速找出“可拆任务、断开的主题、写作素材和临时收件箱”。
- 本轮计划：把笔记页“沉淀雷达”从统计卡改成可点击筛选器，支持 `focus=tasks/links/writing/inbox`，点击后左侧列表直接收束到对应材料。
- 功能边界：不新增数据库表、不新增设置项、不做复杂标签系统；继续基于已有正文清单、`[[双链]]`、folder 和 tags 判断。
- UI 调整：雷达卡增加选中态和“筛选中”标记，列表标题随当前筛选变化，笔记链接保留当前筛选上下文。
- 产品价值：减少周报、组会和写作前从一堆笔记里翻材料的时间，让笔记工作室更像“材料收束台”，而不是普通 Markdown 列表。

## 第七十二轮落地：任务转实验候选队列

- 调研依据：理工科课题推进里，很多“下一步任务”最终都要落成实验、对照、复现、数据采集或指标验证。用户不应该从全部任务里人工判断哪些适合进入实验日志。
- 本轮计划：项目页新增 `focus=experiment` 筛选，让“可转实验”雷达和快捷视图真正收束到包含实验、复现、对照、数据、指标等关键词的未完成任务。
- 功能边界：不新增表、不做复杂规则引擎、不自动理解任务语义；只基于标题、描述和标签的轻量关键词判断，并复用已有“任务转实验”动作。
- UI 调整：“可转实验”增加绿色信号态，筛选条展示当前“可转实验”状态，搜索表单保留该筛选上下文。
- 产品价值：把课题任务自然推向实验记录，减少“任务写了但没有变成证据”的断点，让项目页更贴合理工科从计划到实验验证的日常流。

## 第七十三轮落地：文献转待验证任务

- 调研依据：Zotero 解决“文献在哪里”，实验日志解决“实验怎么记”，但研究生日常更常见的中间状态是“这篇论文有个方法/指标/对照值得验证，还没到正式开实验”。如果只能转实验，会让很多早期想法被迫进入过重的记录形态。
- 本轮计划：在文献卡片增加“待验证”动作，把单篇文献一键转成项目页的未完成任务，并自动进入“可转实验”筛选队列。
- 功能边界：不新增表、不新增配置、不替代 Zotero 分类；复用现有 Task 表，用隐藏 marker 做幂等去重，避免重复点击生成多条相同任务。
- UI 调整：动作放在“阅读笔记 / 转实验”旁边，按钮文案使用“待验证”，强调研究下一步而不是后台任务管理。
- 产品价值：让文献输入可以轻量流向项目执行，减少“读到一个可试想法但还没准备好写实验”的断点。

## 第七十四轮落地：数据复现清单

- 调研依据：研究数据管理实践强调数据来源、版本、位置、处理说明和依赖结果的可追溯；但个人研究生工作台不应变成重型数据仓库。真正高频的是组会、论文或交接前确认“这份数据还能不能找回、复现和解释”。
- 本轮计划：在数据集卡片增加“复现清单”，一键生成可编辑 Markdown 笔记，汇总数据来源、版本、路径、外部链接、处理说明和依赖结果。
- 功能边界：不新增表、不上传文件、不做权限或数据仓库；复用现有 Dataset、Result 和 Note，用隐藏 marker 去重，重复点击直接打开已有清单。
- UI 调整：数据集卡动作区增加轻量按钮，避免把复现核对做成更多表单字段；清单生成后回到笔记页继续补充。
- 产品价值：让数据集从“登记项”变成可复现证据的锚点，减少组会、写论文或交接前找不到数据版本和依赖结果的摩擦。

## 第七十五轮落地：导师反馈回填

- 调研依据：理工科研究生和博士生的高频沟通不是只在会前准备材料，真正容易断的是会后把导师反馈、风险判断、阅读建议和下周动作拆回项目任务。
- 本轮计划：在事务页“组会/导师周报准备”卡片增加“回填导师反馈”，一键生成可编辑 Markdown 笔记，承接本周期周报草稿、临近任务和近期结果证据。
- 功能边界：不新增表、不做会议系统、不做复杂模板配置；复用 Note、Task 和已有“笔记待办拆任务”能力，生成后用户可直接在笔记页补充并拆回项目页。
- UI 调整：按钮与“生成/继续周报笔记”并列，强调会后 5 分钟回填，不把反馈记录藏在设置或新模块里。
- 产品价值：让组会前准备、会后反馈、项目任务和结果证据形成闭环，减少导师意见散落在微信、纸笔或脑子里的摩擦。

## 第七十六轮落地：今日收口清单

- 调研依据：理工科研究生的研究流通常被逾期任务、久未更新实验、待补结果证据和读了一半的文献打断；首页只展示风险还不够，需要让这些尾巴沉淀成当天可处理的清单。
- 本轮计划：首页“收口提醒”增加“生成收口清单”，自动汇总逾期任务、久未更新任务、久未收口实验、待补结果证据和长期未处理文献，生成可编辑 Markdown 笔记。
- 功能边界：不新增表、不新增配置、不做完整审计或 GTD 系统；复用现有 Task、Experiment、Result、Paper 和 Note。
- UI 调整：按钮放在收口提醒卡右上角，和“状态稳定/需要看一眼”并列，不打断首页主行动。
- 产品价值：让首页从“提醒我哪里有风险”升级为“帮我生成今天能收尾的清单”，减少研究债长期堆积。

## 第七十七轮落地：阅读收口清单

- 调研依据：Zotero 负责文献库，但研究生真正高频的断点是“读中/已读却没有留下可回顾笔记”，组会、开题或写 related work 时很难找回当时读出的价值。
- 本轮计划：文献页“阅读雷达”增加“生成阅读收口清单”，把当前列表中读中/已读且缺少阅读备注的文献整理成一篇可编辑 Markdown。
- 功能边界：不新增表、不托管 PDF、不替代 Zotero；只复用 Paper 和 Note，不自动生成未经确认的学术事实。
- UI 调整：入口放在阅读雷达底部，和“读后补笔记”信号放在同一区域，不打断同步、筛选和批量状态更新。
- 产品价值：把“读完但没沉淀”的文献尾巴收住，后续可继续生成单篇阅读笔记、待验证任务、复现实验或综述矩阵。

## 第七十八轮落地：Zotero 增量同步游标

- 调研依据：Zotero Web API 支持基于库版本的增量读取；对忙碌研究生来说，“每次同步都重新扫一遍库”既慢，也会制造不必要的等待感。文献源头仍应交给 Zotero，研途 Hub 只降低进入阅读队列的成本。
- 本轮计划：同步 Zotero 后保存当前库版本游标；下次同一库/集合范围同步时只请求游标之后更新的条目，并在同步反馈里明确显示“增量同步”。
- 功能边界：不新增数据库表、不新增用户配置、不改 Zotero-first 边界；复用 AppSetting 存储每个库/集合范围的同步游标。若仍有更多分页未拉取，不推进游标，避免漏同步。
- UI 调整：文献页同步成功提示区分首次/全量同步和增量同步；没有新增条目时明确告诉用户“当前范围没有新的或更新过的文献”，不让用户误以为同步失败。
- 产品价值：减少文献页等待时间和心理负担，让用户打开后更快进入“读哪几篇、哪些要收口”的核心流程。

## 第七十九轮落地：笔记来源快捷筛选

- 调研依据：Obsidian 的双链和反链适合长期知识网络，但研究生在周报、组会、开题和论文写作前更常见的动作是按来源快速找材料，例如阅读摘录、实验复盘、导师反馈、结果证据和写作草稿。
- 本轮计划：笔记页新增 `source=reading/experiment/meeting/result/writing` 快捷筛选，和现有 `focus=tasks/links/writing/inbox` 可叠加使用。
- 功能边界：不新增数据库表、不新增配置、不做复杂标签体系；只根据笔记标题、分类、标签和正文关键词轻量识别来源，用户仍可通过分类和搜索精确收束。
- UI 调整：左侧增加“来源快捷筛选”小卡片，保留搜索、分类和沉淀雷达的上下文；列表标题显示当前筛选组合，切换来源不会丢掉当前关键词或分类。
- 产品价值：减少“明明生成过阅读计划、实验复盘、结果清单，却要在一堆笔记里翻”的时间，让笔记工作室更适合组会前和写论文前快速收材料。

## 第八十轮落地：开箱进度不误判

- 调研依据：开源自托管工具第一次打开时最容易把用户推向配置和迁移历史资料；对忙碌研究生来说，更合理的上手路径是先接住一条真实文献、课题、实验或笔记。
- 本轮计划：首页不再把“生成上手清单”当作完成开箱的正式数据；上手清单只是路线图，真正创建文献、课题、任务、实验、结果或有效笔记后才进入日常指挥台。
- 功能边界：不新增数据库表、不新增配置、不引入 onboarding 状态表；用上手清单隐藏标记排除统计，并复用现有数据数量判断启动进度。
- UI 调整：空库首页在已生成上手清单后显示“打开上手清单”，仍保留开箱指挥台；有少量真实数据但尚未跑顺时，在首页显示“启动进度”四步卡片，完成后自动消失。
- 产品价值：避免用户点一次清单就失去新手引导，让第一次部署后的体验更像人类可用的研究工作台，而不是一页填完就进入复杂系统。

## 第八十一轮落地：三篇启动阅读队列

- 调研依据：Zotero 能同步大量文献，但研究生真正需要的是每天少量推进、读完留下笔记和实验线索；阅读入口如果只是一长串列表，会把文献管理压力转移到研途 Hub。
- 本轮计划：文献页新增“三篇启动队列”，从全库待读/读中文献中优先挑 3 篇，不受当前搜索、集合或状态筛选干扰。
- 功能边界：不新增数据库表、不做推荐算法、不分析 PDF；只基于已有阅读状态、备注和更新时间排序，继续保持 Zotero-first。
- UI 调整：Hero 右侧阅读栈和正文启动队列统一展示这 3 篇，并提供“标为读中”“阅读笔记”两个低负担动作，避免用户在文献列表里迷路。
- 产品价值：把“同步了一堆文献不知道先读哪几篇”的选择成本压缩到 3 篇，让阅读自然进入笔记、待验证任务和实验草稿。

## 第八十二轮落地：三项实验收口队列

- 调研依据：电子实验记录工具的核心价值不是让用户维护更多参数，而是让实验目的、观察、结论、失败原因和下一步可追溯。理工科研究生日常最容易断在“实验做了但没有收口成结果或复盘”。
- 本轮计划：实验页新增“三项实验收口”，从全库失败、进行中、已完成但无结果证据的实验中优先挑 3 条，不受当前筛选干扰。
- 功能边界：不新增数据库表、不做复杂 ELN、不做样品/仪器管理；继续复用实验状态、结果证据、复盘笔记和复盘任务。
- UI 调整：Hero 右侧实验栈和正文收口队列统一展示这 3 条，并提供“复盘任务”“结果证据”“复盘笔记”等直接动作。
- 产品价值：减少“实验记录写了但没有变成证据/下一步”的断点，让失败实验、进行中实验和结果页自然连起来。

## 第八十三轮落地：三项课题推进队列

- 调研依据：OSF 这类项目容器适合组织阶段和材料，但个人研究生更常见的问题是“任务很多，今天到底先做哪三个”。项目页应降低选择成本，而不是让用户维护复杂项目管理系统。
- 本轮计划：项目页新增“三项课题推进”，从全库未完成任务中优先挑逾期、今天截止、高优先级、进行中和可转实验任务，不受当前筛选干扰。
- 功能边界：不新增数据库表、不做甘特图、不做多人协作；继续复用任务状态、优先级、截止、里程碑挂载和任务转实验能力。
- UI 调整：Hero 右侧推进栈和正文推进队列统一展示这 3 个动作，并提供“移动状态”“挂到里程碑”“转实验”等低负担动作。
- 产品价值：把课题路线图从任务看板进一步收束成每日可执行入口，减少“看板很完整但不知道先推进什么”的摩擦。

## 第八十四轮落地：三条结果证据缺口

- 调研依据：研究数据管理强调结果可追溯、可复现、可复用；但个人研究生更常见的问题是“指标跑出来了，组会/周报/论文前才发现缺复现、缺图表路径或没整理成素材”。
- 本轮计划：成果页新增“三条结果证据缺口”，从全库结果中优先挑待复现、复现中、待判断、缺图表路径或未标记写作素材的 3 条，不受当前筛选干扰。
- 功能边界：不新增数据库表、不做数据仓库、不上传文件、不替代 MLflow/DVC；继续复用结果证据、数据来源、写作素材和证据任务能力。
- UI 调整：Hero 右侧证据栈和正文缺口队列统一使用全库优先级，卡片直接提供“写作素材”和“变任务”动作，减少用户在长结果列表里找尾巴。
- 产品价值：把结果页从“结果归档列表”进一步收束成每日证据收口入口，减少“有结果但不能讲、不能写、不能复现”的科研断点。

## 第八十五轮落地：三件事务减负队列

- 调研依据：研究生的轻行政压力常来自组会、材料、报销和硬截止的碎片打断；事务页的价值不是替代日历或审批系统，而是把最该处理的小事从脑子里移走。
- 本轮计划：事务页新增“三件事务减负”，从全库未完成事务中优先挑逾期、今天截止、进行中和组会相关事项，不受当前筛选干扰。
- 功能边界：不新增数据库表、不做完整日历、不做审批流、不做多人协作；继续复用事务状态、截止、组会/周报草稿和导师反馈回填能力。
- UI 调整：Hero 右侧减负栈和正文减负队列统一展示全库 3 件，卡片直接提供“开始处理/标记完成”和编辑入口，减少在事务列表里翻找尾巴。
- 产品价值：把轻事务从“又一个待办列表”收束成每日低负担入口，让用户快速处理会打断科研的小事，然后回到文献、实验、结果和写作主线。

## 第八十六轮落地：AI 最近研究材料包

- 调研依据：研究生使用 AI 的高频场景不是闲聊，而是把已有任务、实验、结果、文献和笔记快速整理成组会、周报、复盘或写作草稿；最大的摩擦是每次都要手动复制上下文。
- 本轮计划：AI 页新增“最近材料包”，自动汇总近期任务、实验、结果、文献、笔记和事务，作为可编辑 prompt 预填到助手工作台。
- 功能边界：不做自动 RAG、不静默发送全部数据、不替代人工判断；材料只预填在输入框，用户点击生成前仍可删减敏感内容。
- UI 调整：侧栏增加材料计数卡，Hero 助手栈优先展示“整理今天”；工作台默认打开最近材料包，同时保留组会提纲、实验复盘、阅读卡片和写作润色场景。
- 产品价值：减少“想用 AI 但还要先到处复制材料”的阻力，让 AI 页更像研究工作台的草稿整理器，而不是孤立聊天框。

## 第八十七轮落地：首页三件开工动作

- 调研依据：理工科研究生日常打开工作台时最需要的是“现在先做哪三件”，而不是分别进入项目、事务、实验、成果和文献页再判断优先级。
- 本轮计划：首页第一屏新增跨模块“三件开工动作”，从未完成任务、轻事务、久未更新实验、待补结果证据和待读文献中统一排序。
- 功能边界：不新增数据库表、不做复杂推荐算法、不替代各模块专业页面；只复用现有截止、状态、更新时间和结果证据字段进行轻量排序。
- UI 调整：Hero 主标题和行动栈优先展示跨模块开工动作，行动栈可直接点击跳转到对应页面；没有数据时仍保留快速捕捉式空状态。
- 产品价值：把首页从“研究信号汇总”进一步收束成“今天开工入口”，降低用户每天在多个模块之间来回判断的认知成本。

## 第八十八轮落地：Zotero 连接三步引导

- 调研依据：Zotero API 能解决文献重复录入问题，但自托管用户第一次连接时最容易卡在 API Key、Library ID、Library Type 和 Collection Key 的概念上。
- 本轮计划：设置中心新增“Zotero 三步连接”说明卡，把创建只读 Key、填写个人/群组 Library ID、按需使用 Collection Key 讲清楚。
- 功能边界：不新增数据库表、不新增配置字段、不改变 Zotero-first 策略；只把现有连接项翻译成人类能看懂的步骤。
- UI 调整：说明卡放在设置中心左侧，与当前状态和备份并列；外链指向 Zotero 官方设置页，错误提示集中说明常见填错点。
- 产品价值：降低首次部署后的连接门槛，让用户尽快从“配置文献库”进入“同步后开始阅读”，避免把文献管理变成新的维护负担。

## 第八十九轮落地：文献页首次同步引导

- 调研依据：文献页是很多研究生部署后第一个会打开的核心页面；如果只显示“去设置”，用户仍然不知道为什么要接 Zotero、接完后能得到什么。
- 本轮计划：文献页在 Zotero 未连接时展示三步上手卡，直接说明只读 Key、Library ID 和同步后的“三篇启动队列”。
- 功能边界：不新增配置项、不改变同步逻辑、不鼓励手动维护第二套文献库；手动补录只作为临时材料兜底入口。
- UI 调整：用低饱和暖砂提示区替代单行警告，主操作是“去设置连接”，次操作是创建 Zotero Key，底部保留“临时补录”按钮。
- 产品价值：让第一次进入文献页的用户快速理解 Zotero-first 工作流，减少重复录文献和配置迷路的概率。

## 第九十轮落地：实验模板即入口

- 调研依据：电子实验记录工具的高频价值在于用模板减少重复书写，让用户快速进入目的、方法、观察、结论和下一步；个人工作台不应把模板藏在表单内部。
- 本轮计划：实验页左侧模板卡直接变成“使用模板”入口，点击即可打开新建实验弹窗并预选对应模板。
- 功能边界：不新增字段、不做复杂 ELN、不引入样品/仪器/权限等重型能力；继续复用现有实验表单和模板正文。
- UI 调整：模板卡保留简短说明，右侧增加小按钮；用户从“想做复现/消融/排错”直接进入对应记录纸。
- 产品价值：减少新建实验时的选择成本，让忙碌研究生可以用一秒钟选定记录结构，而不是在弹窗里再找模板。

## 第九十一轮落地：30 秒加任务

- 调研依据：研究生项目管理的高频需求不是维护完整项目管理字段，而是随手收住下一步动作，避免导师反馈、文献想法和实验尾巴散掉。
- 本轮计划：项目页侧栏新增“30 秒加任务”，只填任务标题、优先级、截止和可选里程碑，即可进入任务队列。
- 功能边界：不新增数据库字段、不替代完整任务表单、不做复杂 GTD；轻量入口复用现有 `Task` 和 `createTask`。
- UI 调整：入口放在项目页侧栏第一张卡，完整“新建任务”弹窗仍保留给需要补上下文的情况。
- 产品价值：降低记录下一步的阻力，让任务页更像忙碌研究生的随手推进台，而不是需要认真填写的项目管理系统。

## 第九十二轮落地：笔记快速摘记

- 调研依据：研究生的很多有效材料来自临时想法、导师一句反馈、实验观察或文献摘录；如果每次都进入完整编辑器，会降低记录意愿。
- 本轮计划：笔记页侧栏新增“快速摘记”，只填一句标题和正文，默认保存到收件箱。
- 功能边界：不新增表、不引入复杂编辑器、不替代右侧沉浸式笔记；轻量入口复用现有 `Note` 和 `createNote`。
- UI 调整：快速摘记放在写作素材提示之后、沉淀雷达之前，避免占用右侧主编辑区；正文保留 4 行输入，适合随手收住材料。
- 产品价值：让笔记页同时支持“沉浸写作”和“快速收件箱”，减少碎片材料流失。

## 第九十三轮落地：60 秒记录证据

- 调研依据：理工科研究生常常先得到一条关键指标或图表，再慢慢补实验、数据集和复现说明；成果页如果一开始就要求完整证据表单，会降低记录意愿。
- 本轮计划：成果页侧栏新增“60 秒记录证据”，只填一句话结果、一个核心指标、复现状态、图表路径和简短结论。
- 功能边界：不新增表、不做数据仓库、不上传文件；轻量入口复用现有 `Result` 和 `createResult`，完整结果弹窗继续保留。
- UI 调整：入口放在成果页侧栏第一张卡，用低饱和纸面样式强调“先收住证据，后补关联”。
- 产品价值：让跑出指标后能立刻进入证据台，减少结果散落在截图、表格、聊天记录和脑子里的情况。

## 第九十四轮落地：30 秒登记小事

- 调研依据：组会、材料、报销和硬截止会频繁打断研究节奏；个人工作台应该先帮用户把小事移出脑子，而不是要求完整填写事务表单。
- 本轮计划：事务页侧栏新增“30 秒登记小事”，只填标题、类型、截止、地点/渠道和简短备注。
- 功能边界：不新增表、不做完整日历或审批流；轻量入口复用现有 `AdminItem` 和 `createAdminItem`，完整事务弹窗继续保留。
- UI 调整：入口放在事务页侧栏第一张卡，默认状态为待处理、标签为快速事务。
- 产品价值：降低记录轻事务的门槛，让用户快速清空脑内负担，再回到文献、实验、结果和写作主线。

## 第九十五轮落地：AI 草稿流降噪

- 调研依据：研究生使用 AI 的高频动作是把已有材料压缩成组会、复盘、阅读或写作草稿；如果页面同时堆场景、配置、模型细节和长说明，会让 AI 页变成另一个配置台。
- 本轮计划：AI 页左侧改为“材料包 → 生成草稿 → 保存笔记”的三步草稿流，场景按钮集中放到右侧工作台顶部。
- 功能边界：不新增数据库表、不新增 AI 能力、不自动读取或发送全部数据；材料包仍只是预填 prompt，用户提交前可删改。
- UI 调整：删除左侧重复场景卡，模型信息折叠到“查看当前模型”；右侧增加“恢复材料包”和“清空”按钮，并把输入自动保存到当前浏览器。
- 产品价值：减少页面噪声和误操作成本，让 AI 页更像科研草稿工作台，而不是聊天应用或后台配置面板。

## 第九十六轮落地：快速捕捉免记忆

- 调研依据：Zotero、Obsidian、ELN 和任务工具的共性启发是入口要足够低摩擦；研究生最常丢失的是导师一句话、实验现象、结果数字和下一步动作，不应该先记住系统语法。
- 本轮计划：顶部快速捕捉增加中文示例和可点击快捷 chip，让用户不用记前缀也能把一句话分流到任务、实验、结果、文献、组会或想法。
- 功能边界：不新增配置、不新增字段、不改变快速捕捉解析逻辑；示例只调用已有 `quickCapture` 分流能力。
- UI 调整：输入框 placeholder 改成更短的人类提示，补充 `datalist` 示例；桌面端展示低饱和快捷 chip，移动端保持简洁不挤占空间。
- 产品价值：降低第一次使用和每天随手记录的门槛，让“想到就收住”成为默认动作，而不是打开某个模块再填表。

## 第九十七轮落地：快速捕捉示例先填入

- 调研依据：低摩擦不等于无确认；研究生快速记录时最怕误点后生成一条脏数据，示例入口应该负责教学和预填，而不是直接替用户提交。
- 本轮计划：把顶部快捷 chip 从“点击即创建”改成“点击填入输入框”，用户确认内容后再点捕捉。
- 功能边界：不改 `quickCapture` 服务端分流逻辑、不新增撤销系统、不改变移动端布局；只把输入框和示例 chip 拆成轻量 client 组件。
- UI 调整：保留 datalist 和桌面端示例 chip，chip 标题改为“填入示例”，输入框受控显示当前示例内容。
- 产品价值：在保持学习成本低的同时减少误造记录，让首页入口更适合长期日常使用。

## 第九十八轮落地：开箱页第一条真实记录

- 调研依据：第一次部署后最容易卡在“先配置什么、先填哪个模块”；理工科研究生更需要一个能立刻接住任务、实验现象、结果数字或想法的入口。
- 本轮计划：在空库开箱页增加“先写第一条真实记录”区域，直接复用快速捕捉输入和示例。
- 功能边界：不新增 onboarding 状态、不新增数据模型、不改变顶部快速捕捉；只把现有 `QuickCaptureBar` 复用到开箱页。
- UI 调整：把快速捕捉放在研究流和四个 StartCard 之间，用低饱和纸面卡片强调“示例先填入，确认后提交”；同时为 `datalist` 使用唯一 id，避免页面多处复用时冲突。
- 产品价值：让新用户从第一分钟就能写入真实科研流，减少只看引导却没有产生有效记录的情况。

## 第九十九轮落地：快速捕捉科研语义扩展

- 调研依据：Zotero 负责文献集合和标签，Obsidian 强在碎片笔记与关联，ELN 强在实验记录；研途 Hub 的快速入口应该接住导师反馈、卡点、实验观察和复盘，而不是只识别“任务/实验/文献”这些系统词。
- 本轮计划：扩展快速捕捉示例和前缀别名，让“导师/反馈/卡点/问题/待确认”进入任务，“观察/现象/复盘/失败”进入实验记录。
- 功能边界：不新增字段、不新增 AI 判断、不做复杂自然语言分类；继续使用显式前缀，保持可预期。
- UI 调整：顶部和开箱页的示例 chip 改成更贴近研究生日常语言，并全部展示，不再截断到 5 个。
- 产品价值：用户可以用自己平时说话的方式捕捉研究碎片，减少“我该写任务还是笔记”的犹豫。

## 第一百轮落地：快速捕捉去向预览

- 调研依据：显式前缀虽然可预期，但用户提交前仍需要知道这句话会进入哪个模块；低负担入口必须同时降低误投递和脏数据成本。
- 本轮计划：在快速捕捉输入框下方增加“将保存到”预览，实时显示任务、实验、成果、文献、事务、数据或笔记去向。
- 功能边界：不改变服务端 `quickCapture` 权威分流逻辑、不新增接口、不做 AI 分类；客户端只复刻前缀映射用于提示。
- UI 调整：空输入时显示“等待输入”，输入后展示去向和一句简短说明；保留示例 chip 的填入交互。
- 产品价值：用户在点击捕捉前就能判断是否会投到正确模块，减少误用和后续整理成本。

## 第一百零一轮落地：快速捕捉规则共享

- 调研依据：快速入口的前端预览如果和服务端分流规则分开维护，后续新增前缀时容易出现“显示进入实验，实际进了笔记”的错位。
- 本轮计划：抽取 `src/lib/quick-capture.ts`，统一维护示例、前缀解析、别名映射和去向标签。
- 功能边界：不改变现有捕捉行为、不新增 UI、不新增数据库字段；只是把重复规则收敛成共享模块。
- UI 调整：QuickCaptureBar 从共享模块读取示例和去向说明；服务端 action 从同一模块读取前缀解析和别名映射。
- 产品价值：保证“用户看到的去向”和“实际保存去向”长期一致，降低维护成本和误投递风险。

## 第一百零二轮落地：三篇阅读计划入口

- 调研依据：Zotero 同步后最容易产生长列表压力；文献页已经有“三篇启动队列”，但用户还需要一个只针对这三篇的行动入口，而不是再从全列表里挑选。
- 本轮计划：在“三篇启动队列”标题区新增“生成三篇计划”按钮，只把当前启动队列的 1-3 篇文献传给阅读计划生成逻辑。
- 功能边界：不新增推荐算法、不改变 Zotero 数据、不新增字段；复用现有 `createReadingPlanNote`。
- UI 调整：按钮放在全库待处理数量旁边，视觉上和队列绑定，避免用户误以为会处理当前长列表全部文献。
- 产品价值：把“先读哪几篇”进一步落到一篇可勾选阅读计划，减少同步后不知道怎么开始的摩擦。

## 第一百零三轮落地：三项实验收口清单

- 调研依据：理工科研究生的实验压力通常来自“做了但没收口”，尤其是失败、久未更新或缺结果证据的实验；ELN 的价值应体现在复盘和下一步，而不是字段完整度。
- 本轮计划：实验页“三项实验收口”标题区新增“生成收口清单”按钮，只针对当前自动挑出的 1-3 条实验生成当天可编辑笔记。
- 功能边界：不新增数据库字段、不新增复杂实验管理、不替代单条实验复盘；复用 `Note` 作为临时收口工作纸。
- UI 调整：入口与“三项实验收口”队列绑定，和全库待收口数量放在同一操作区，避免用户误以为会处理全部实验。
- 产品价值：把“这几个实验该补什么”落成一张可勾选清单，降低失败实验、结果证据和下一步任务之间的切换成本。

## 第一百零四轮落地：今日导师沟通单

- 调研依据：项目管理工具强调任务/看板，ELN 强调实验追踪，MLflow 强调结果指标，但研究生日常最常见的临时压力是“导师突然问进展”或“组会前几分钟需要压缩材料”。
- 本轮计划：事务页在“导师沟通准备度”和周报卡片中新增“今日沟通单”，复用现有组会草稿 action，但按今日范围生成更短的导师沟通模板。
- 功能边界：不新增表、不增加配置、不做会议系统；继续用 `Note` 保存沟通单，周报草稿仍保留给每周汇总。
- UI 调整：今日沟通单作为主按钮，周报草稿降为次按钮；已有今日沟通单时直接打开，不重复生成。
- 产品价值：把任务、实验、结果、文献和事务压缩成当天可讲的 3 分钟版本，降低临时沟通和会前准备成本。

## 第一百零五轮落地：三条结果证据收口清单

- 调研依据：MLflow Tracking 强调记录 runs 的 metrics、parameters 和 artifacts，DVC 强调数据/实验版本与结果可追溯；个人研究生不需要把这些系统搬进日常界面，但需要把结果指标、复现状态、图表路径和一句话结论收住。
- 本轮计划：成果页“三条结果证据缺口”标题区新增“生成证据清单”按钮，只针对当前自动挑出的 1-3 条缺口结果生成当天可编辑收口笔记。
- 功能边界：不新增表、不引入 MLflow/DVC 配置、不上传文件；继续复用 `Result`、`Note` 和现有待补任务/写作素材动作。
- UI 调整：入口与证据缺口队列绑定，和“全库待补”数量放在同一操作区，避免用户误以为会整理全部结果。
- 产品价值：把“这条结果还能不能讲”变成一张可勾选清单，降低组会、周报和论文写作前找指标、找图、找复现状态的成本。

## 第一百零六轮落地：今日笔记沉淀清单

- 调研依据：Obsidian 的内部链接和 backlinks 证明笔记价值在上下文关系，但个人研究生不需要维护完整知识图谱；更高频的是把收件箱、待办清单、导师反馈和可写作素材及时收口。
- 本轮计划：笔记页“今日沉淀栈”新增“生成清单”按钮，只针对当前 1-3 篇最值得处理的笔记生成当天可编辑沉淀笔记。
- 功能边界：不新增表、不做知识图谱、不自动创建双链主题；继续复用 `Note`、现有 checklist 拆任务和写作素材包。
- UI 调整：按钮放在深色沉淀栈右上角，视觉上绑定当前推荐队列，避免用户误以为会整理全部笔记库。
- 产品价值：把“这篇笔记下一步该怎么处理”落成清单，减少碎片长期停在收件箱或笔记里藏着待办却没有回到项目页的情况。

## 第一百零七轮落地：三项课题推进清单

- 调研依据：OpenProject、GitLab Issues 等项目管理工具适合多人协作和完整看板，但个人研究生最高频需求是“今天先推进哪三件”，而不是维护复杂项目字段。
- 本轮计划：项目页“三项课题推进”标题区新增“生成三项清单”按钮，只把当前自动挑出的 1-3 个任务交给现有课题推进笔记 action。
- 功能边界：不新增表、不做甘特图、不改变任务排序算法；复用 `createProjectProgressNote` 和现有任务/里程碑关系。
- UI 调整：按钮与全库待推进数量放在一起，强调只处理当前推荐队列，避免把长任务列表变成新的压力源。
- 产品价值：让用户从长任务列表直接得到一张可编辑推进清单，先确认本周/今天真正要推进的动作，再把能实验化的任务转到实验日志。

## 第一百零八轮落地：快速捕捉键盘入口

- 调研依据：Zotero、Obsidian、ELN 和任务工具共同强调低摩擦记录入口；研究生最常丢的是导师反馈、实验现象、结果数字和下一步动作，入口不应要求先切页面或找按钮。
- 本轮计划：顶部快速捕捉增加 `Ctrl/Cmd + K` 和 `/` 聚焦入口，输入时 `Esc` 可清空，继续显示保存去向预览。
- 功能边界：不新增数据表、不改变快速捕捉服务端分流规则、不做全局命令面板；只是让现有一句话捕捉更接近日常工作流。
- UI 调整：输入框右侧增加紧凑快捷键提示，帮助文本中说明聚焦和清空方式；移动端不额外增加拥挤元素。
- 产品价值：让用户在任何页面都能快速收住研究碎片，减少“想到一件事但还要先找模块”的中断成本。

## 第一百零九轮落地：快速捕捉防误提交

- 调研依据：低摩擦入口也要避免制造脏数据；研究生随手记录时常会误点按钮或空输入提交，后续再清理比一开始拦住更费时间。
- 本轮计划：快速捕捉空输入时禁用提交按钮，输入框内支持 `Ctrl/Cmd + Enter` 直接提交，继续保留 `Esc` 清空。
- 功能边界：不改变服务端解析和落库逻辑，不新增撤销系统；只在客户端交互层减少误提交和鼠标操作。
- UI 调整：帮助文案补充提交快捷键，按钮禁用态沿用现有设计系统，不额外增加说明卡。
- 产品价值：让“想到就写、确认就收”的路径更快，同时减少空记录和误操作带来的整理负担。

## 第一百一十轮落地：导航即研究闭环说明

- 调研依据：Zotero、ELN、Obsidian 和 RDM 工具各自强项不同，新用户第一次打开个人工作台时最容易困惑的是“我该进哪个页面”，而不是缺少更多功能。
- 本轮计划：桌面侧边栏每个导航项增加一句极短用途说明，让首页、文献、实验、课题、笔记、成果、事务、AI 和设置的边界更清楚。
- 功能边界：不新增页面、不新增配置、不改变路由；移动端仍保持紧凑标签，避免小屏过载。
- UI 调整：选中态继续保持深色标识，说明文字使用低对比辅助色，服务快速扫视而不是制造信息噪音。
- 产品价值：降低新用户学习成本，让导航本身解释科研闭环，减少“这个功能到底该放在哪里”的犹豫。

## 第一百一十一轮落地：开箱先写真实记录

- 调研依据：第一次部署后的最大摩擦不是缺少功能，而是用户被设置、Key、历史迁移吓住；理工科研究生更需要先把当前真实研究流写进去，再按需连接 Zotero 或 AI。
- 本轮计划：空库开箱页把第一行动从“完成设置中心”改为“写第一条真实记录”，设置中心降级为“可选连接”。
- 功能边界：不新增 onboarding 状态、不新增配置、不改变快速捕捉和设置逻辑；只调整开箱页的信息顺序。
- UI 调整：四张上手卡改成“真实记录 -> Zotero -> 课题 -> 可选连接”，检查清单也把 Key 配置放到最后。
- 产品价值：让新用户第一分钟先产生可复盘的研究数据，而不是先进入配置心态，符合“人类使用而不是机器维护”的目标。

## 第一百一十二轮落地：开箱反配置原则卡

- 调研依据：研究生迁移工具时最怕“又多一个要维护的系统”；清楚告诉用户暂时不用搬 PDF、旧实验和不常用配置，比继续增加功能更能降低上手焦虑。
- 本轮计划：空库开箱页“暂时不用做”从普通段落改成三条反配置原则卡，强调不搬旧资料、不堆配置项、不追求一次完美。
- 功能边界：不新增功能、不新增字段、不改业务逻辑；只优化开箱页的信息呈现和视觉层级。
- UI 调整：使用低饱和虚线卡和暖砂图标，和主行动卡区分，不制造新的操作入口。
- 产品价值：让新用户明确这个平台是为了节省时间，而不是要求他们先完成一次大迁移或配置工程。

## 第一百一十三轮落地：文献接入去配置化

- 调研依据：Zotero 已经负责文献库、附件和引用信息，研途 Hub 的价值是同步后帮用户决定先读哪三篇；文献页不应把用户推向手动补录或长配置说明。
- 本轮计划：文献页 Zotero 未连接提示进一步压缩为“建只读 Key -> 填 Library ID -> 同步后只读三篇”，并强调同步前只做接上 Zotero 这一件事。
- 功能边界：不改变 Zotero 同步逻辑、不新增配置项、不新增文献字段；手动补录仍只作为临时材料兜底。
- UI 调整：三步卡标题缩短，底部提示明确“正式文献回 Zotero，临时材料才补录”，减少重复维护两套文献库的暗示。
- 产品价值：降低第一次进入文献页的配置压力，让用户从“我该填什么字段”转为“接上 Zotero 后先读三篇”。

## 第一百一十四轮落地：实验入口去后台化

- 调研依据：ELN 的高频价值是快速留下目的、观察、结论和下一步，而不是让个人研究生感觉自己在新建一条后台数据。
- 本轮计划：实验页主按钮从“新建实验”改为“记录实验”，弹窗标题改为“记录一次实验”，描述强调先写目的、关键观察和下一步。
- 功能边界：不改实验表单、不新增模板、不新增字段；只调整高频入口文案和心理模型。
- UI 调整：保留弹窗交互和现有模板能力，弱化“参数/附件必须补全”的暗示。
- 产品价值：降低记录实验的启动阻力，让用户把实验页当成研究日志，而不是参数台账。

## 第一百一十五轮落地：课题入口去项目管理化

- 调研依据：OSF、OpenProject、GitLab Issues 等工具适合完整项目容器或协作看板，但个人研究生最高频需求是把当前课题拆成可推进阶段和下一步动作。
- 本轮计划：课题页 Hero 入口从“新建项目/新建里程碑/新建任务”改为“建课题/拆阶段/记下一步”，弱化后台项目管理语气。
- 功能边界：不改项目、里程碑、任务表单，不新增字段，不做甘特图或多人协作。
- UI 调整：弹窗标题同步改成“建立一个课题”“拆一个阶段”“记下一步动作”，描述强调可验收和可推进。
- 产品价值：让课题页更像研究路线图，而不是通用项目管理系统，降低记录下一步动作的心理阻力。

## 第一百一十六轮落地：暖纸低疲劳视觉与入口去后台化

- 调研依据：Zotero Web API 适合继续作为文献源头；eLabFTW 的实验模板说明高频记录应减少重复书写；OSF components 说明研究项目可以按阶段拆分；Obsidian backlinks 说明笔记价值在关联；RDM 研究强调数据组织、文档化、版本和保存仍是研究生薄弱环节。
- 本轮计划：不新增功能、不新增配置，先把全局视觉从偏冷后台色改成暖纸低疲劳工作台，并把成果、事务两个高频入口继续改成人类化动作。
- 功能边界：不改数据库、不做完整 ELN/OSF/MLflow/Obsidian 复制品，不引入新依赖；只改全局视觉 token、导航壳和入口文案。
- UI 调整：背景改成低对比暖纸层、石板蓝灰主色、鼠尾草绿行动色；卡片阴影减轻；侧边栏品牌语改为“轻量科研行动台”；成果入口改为“记录证据”，事务入口改为“记一件小事”。
- 产品价值：减少长时间使用的视觉疲劳和后台管理感，让用户打开页面后看到的是下一步研究行动，而不是又一套需要维护的系统。

## 第一百一十七轮落地：文献与笔记细节减噪

- 调研依据：Zotero 已经承担文献采集、集合和标签管理；Obsidian 的价值在持续书写和链接关系。研途 Hub 应让文献页聚焦“同步、只读三篇、留下笔记抓手”，让笔记页像写作台而不是表单。
- 本轮计划：弱化文献页手动补录入口，把侧栏技术同步说明改为三步阅读工作流；笔记页主按钮从“新建笔记”改成“写笔记”，编辑区从冷色表单背景改成暖纸写作区。
- 功能边界：不改 Zotero 同步逻辑、不改笔记模型、不新增编辑器依赖；只优化文案、侧栏说明和局部视觉。
- UI 调整：文献集合选中态改成鼠尾草绿系；阅读节奏卡加入 01/02/03 流程；笔记编辑/预览/双链区域统一暖纸底色，保存按钮改为“收进笔记”。
- 产品价值：降低文献页的配置感和笔记页的表单感，让研究生更愿意在长时间阅读和写作时停留。

## 第一百一十八轮落地：实验到证据闭环减噪

- 调研依据：eLabFTW 等 ELN 工具说明实验记录的价值在可追踪正文、状态和链接，但个人研究生不需要完整参数仓库。高频痛点是“实验有观察，却没有收成能汇报的结果证据”。
- 本轮计划：实验页不新增字段，只把“记录结果”改成“收一条证据”，并在侧栏加入“写观察 -> 判结果 -> 回填正文”的三步提示。
- 功能边界：不改 `Experiment` / `Result` 模型，不改 `createResultFromExperiment` 逻辑，不新增图表或上传能力。
- UI 调整：实验侧栏新增“实验收口三步”；实验卡片里的结果区域改为“证据回流”，空状态按钮改成“收一条证据”。
- 产品价值：让实验记录自然流向成果证据台，减少用户在实验页和成果页之间反复判断入口的时间。

## 第一百一十九轮落地：结果证据可讲度看板

- 调研依据：FAIR/RDM 强调数据和结果要可查找、可追溯、可复用；但研究生组会前最急的是“这条结果能不能讲、缺复现还是缺素材”。成果页应优先给出可讲度判断，而不是只展示结果列表。
- 本轮计划：不新增数据表，复用复现状态、图表路径和写作素材标记，在成果页侧栏增加“证据可讲度”四步看板。
- 功能边界：不改结果录入逻辑、不新增指标算法、不做复杂评分；只用现有字段推导“可以讲/先判定/补可信度/补素材”。
- UI 调整：侧栏新增 01-04 可讲度卡片，点击直接切换现有筛选；快捷视图选中态统一到鼠尾草绿系。
- 产品价值：让用户在组会、周报或写论文前快速知道最该处理哪类结果，减少在长列表里反复判断证据状态。

## 第一百二十轮落地：首页今日行动流

- 调研依据：研究生高频痛点不是缺少模块，而是每天开工前不知道先处理任务、文献、实验、证据还是组会材料。首页第一屏应像“今日科研指挥台”，用最少判断给出当天行动顺序。
- 本轮计划：不新增查询和模型，复用现有 `openingActions`、文献队列、实验状态和结果素材计数，把首页右侧从普通信号列表改成“今天只走四步”行动流。
- 功能边界：不做日历、不做提醒系统、不改快速捕捉；只调整首页首屏布局和文案。
- UI 调整：右侧卡片标题改为“今天只走四步”，使用 01-04 顺序：定下一步、读/验证、收口实验、整理证据；每步直接跳到对应模块。
- 产品价值：减少用户打开平台后的选择成本，让首页更像研究节奏引导，而不是统计仪表盘。

## 第一百二十一轮落地：首页去重降噪

- 调研依据：Zotero、eLabFTW、OSF、Obsidian 和 RDM 资料都指向同一个结论：个人科研工作台应该接住连续工作流，而不是把同一组信号重复展示成多块仪表盘。研究生打开首页后需要少判断、少滚动、少扫重复卡片。
- 本轮计划：保留首屏“今天只走四步”和“3 分钟自动整理”，删除主态中重复解释研究节奏的横向节奏带，以及和首屏数字重复的四张统计信号卡。
- 功能边界：不新增数据库表、不新增查询、不改变开箱页教学节奏；只调整主态首页的信息密度，并让开箱态复用自动整理区的引导模式。
- UI 调整：开箱态的自动整理区改为“开箱后会自动整理”，空入口显示“等任务/文献/实验/结果/笔记进入”，避免用户误以为要先配置完整系统。
- 产品价值：首页更像当天行动入口，而不是后台总览；减少视觉疲劳和重复信息，让用户更快进入任务、文献、实验、证据或笔记。

## 第一百二十二轮落地：文献页 Zotero-first 主路径收敛

- 调研依据：Zotero Web API 已经覆盖库、集合、标签和条目同步，文献页不应鼓励用户重复维护第二套文献库；研究生真正高频的是同步后快速决定先读哪三篇、读完留下什么抓手。
- 本轮计划：文献页第一屏只保留“接上/同步 Zotero”和“生成三篇计划”两个主路径；临时补录降级到侧栏兜底入口，并用连接状态卡说明“正式文献回 Zotero”。
- 功能边界：不改 Zotero 同步逻辑、不新增配置项、不新增文献字段；只调整入口层级、侧栏说明和阅读状态拆分。
- UI 调整：侧栏新增“Zotero 到阅读”状态卡，阅读进度单独成卡；未连接时首屏只显示“接上 Zotero”，避免和临时补录、设置、计划按钮并列造成选择压力。
- 产品价值：降低第一次进入文献页的误解和维护成本，让用户从“我要填哪些字段”转向“同步后先读三篇”。

## 第一百二十三轮落地：实验页记录纸入口

- 调研依据：ELN 的高频价值在于用模板降低重复书写，让实验留下可追踪正文、状态和结论；个人研究生不需要参数仓库感，而需要快速开始写目的、观察、结论和下一步。
- 本轮计划：实验页首屏主操作从“记录实验 + 收一条证据”收敛为“选记录纸开写 + 生成三项收口”，模板区改成可直接开写的记录纸入口。
- 功能边界：不新增模板、不新增字段、不改变实验和结果关联逻辑；只调整入口层级、模板卡样式和表单文案。
- UI 调整：模板卡使用暖纸记录纸样式，按钮文案改为“用这张记录纸”；实验表单保存按钮改成“保存到实验日志”，降低后台建表感。
- 产品价值：让实验页更像忙碌研究生随手打开的实验日志，而不是需要先补齐项目/参数/证据的后台表单。

## 第一百二十四轮落地：课题页下一步优先

- 调研依据：OSF components 和项目管理工具适合组织完整研究结构，但个人研究生日常最高频不是维护结构，而是把导师反馈、实验尾巴和论文想法收成下一步动作。
- 本轮计划：课题页首屏从“建课题 / 拆阶段 / 记下一步 / 生成笔记”收敛为“先记下一步”和“生成三项清单”；建课题、拆阶段移到侧栏“课题结构”卡。
- 功能边界：不改项目、里程碑、任务模型，不新增看板或甘特图；只调整入口层级和任务文案。
- UI 调整：侧栏新增“课题结构”卡说明结构不用每天维护；快速任务卡标题改为“30 秒记下一步”，提交按钮改为“收进推进栈”。
- 产品价值：让项目页更像课题推进台，而不是通用项目管理后台；用户打开后先记录或处理下一步，不被结构维护打断。

## 第一百二十五轮落地：笔记页沉浸写作台

- 调研依据：Obsidian 的双链和 backlinks 适合长期知识连接，但研究生高频痛点不是维护完整知识图谱，而是把阅读摘录、实验观察、导师反馈和结果证据快速沉淀为可写材料。
- 本轮计划：笔记页首屏主路径收敛为“继续当前笔记 / 写新笔记 / 生成素材包”，弱化重复入口；侧栏快速摘记默认折叠，避免一进页面就被表单占满。
- 功能边界：不新增编辑器依赖、不改 `Note` 模型、不自动创建双链主题；继续使用现有 Markdown、双链识别、checklist 拆任务和写作素材包能力。
- UI 调整：右侧编辑区改成更稳定的暖纸写作台，整体高度拉满；侧栏从“沉淀雷达 / 来源快捷筛选”改成“整理视角 / 材料来源”，减少后台工具感。
- 产品价值：让笔记页更像长时间可停留的研究写作桌，而不是左侧列表加右侧表单；用户可以先写、后整理，降低记录碎片材料的心理阻力。

## 第一百二十六轮落地：首页高频研究通道

- 调研依据：Zotero Web API 覆盖文献条目、集合、标签和增量同步；eLabFTW 的实验页证明 ELN 高频价值在实验正文、模板、状态和链接；OSF components 强调按研究阶段/材料类型拆分；Obsidian internal links/backlinks 证明笔记价值在上下文关系；RDM 研究强调数据组织、文档化和复用仍是研究生薄弱环节。
- 本轮计划：把首页右侧“研究闭环”从普通列表升级为“高频研究通道”，直接呈现文献、实验、结果、写作四条路径，让用户知道每天最常走的不是完整后台菜单，而是从文献到实验、证据和写作的连续流。
- 功能边界：不新增数据库查询、不引入新依赖、不复制 Zotero/eLabFTW/OSF/Obsidian 的完整能力；只把已有数据汇总换成更清晰、更吸引人的首页入口。
- UI 调整：全局背景去掉细网格，改成更柔和的渐变光斑纸面；卡片透明度和阴影降低；新增 `research-lane-card`，每条通道有低饱和图标、用途标签和直接动作按钮。
- 产品价值：首页更像研究驾驶舱，而不是后台总览。用户打开后能快速进入阅读、实验、证据和写作，不需要先理解所有模块。

## 第一百二十七轮落地：全局研究流导航

- 调研依据：研究生长期使用工作台时，导航不仅是页面入口，也是对产品心智的解释。若侧栏像后台菜单，用户会把平台理解成又一套需要维护的系统；若侧栏直接呈现文献输入、实验验证、结果证据、写作输出，用户更容易按科研流使用。
- 本轮计划：重构全局应用壳的侧栏第一屏，把品牌区、研究流卡片、导航文案和底部今日提醒统一成低疲劳研究驾驶舱，不新增功能，只减少后台感。
- 功能边界：不改路由、不改认证、不改快速捕捉逻辑；只调整 `AppShell`、导航选中态和相关 CSS token。

- 产品价值：用户每天进入任意页面都能看到“文献 -> 实验 -> 结果 -> 写作”的主线，减少模块迷路和后台维护感。

## 第一百二十八轮落地：登录入口去后台化

- 调研依据：开源自托管项目的第一眼通常是登录页。如果登录页像普通后台系统，用户会预期后续也是配置和表格；研途 Hub 应在登录前就传达“个人研究流工作台”的心智，同时不能泄露任何工作台数据。
- 本轮计划：重构登录页左侧品牌区和右侧密码卡，保留单密码登录，不引入多用户或复杂权限；登录前只展示产品流程，不展示用户数据。
- 功能边界：不改认证逻辑、不改密码来源、不新增配置项；只调整视觉、文案和登录表单层级。
- UI 调整：登录页变成半透明纸面驾驶舱，左侧展示文献输入、实验验证、结果证据、写作输出四步；右侧说明访问密码和设置中心职责，错误提示改成轻量卡片。
- 产品价值：第一次访问服务器时不再像后台登录框，而是明确这是为研究生节省时间的私人科研工作台。

## 第一百二十九轮落地：成果页证据用途强化

- 调研依据：RDM/FAIR 强调结果要可追溯、可复用、可解释，但个人研究生在组会、周报和论文草稿前最急的是判断“这条结果能不能讲、还缺什么证据”。
- 本轮计划：成果页继续收敛成“结果证据台”，弱化数据仓库感，把主标题、按钮、筛选和标签页都围绕“能不能讲 / 能不能写”组织。
- 功能边界：不新增字段、不新增配置、不改变结果和数据来源模型；只调整文案、入口权重和局部布局。
- UI 调整：主按钮突出“记录证据”，数据来源降级为辅助入口；“生成汇报清单”改为“收成汇报清单”；移除重复的证据雷达卡，将快捷筛选压缩为两列；“数据集”统一改为“数据来源”。
- 产品价值：用户打开成果页时先看到可执行判断，而不是维护数据后台；结果记录能自然流向组会、周报和论文素材。

## 第一百三十轮落地：首页调研需求罗盘

- 调研依据：Zotero 适合继续承担文献库、集合和标签源头；eLabFTW 类 ELN 的高频价值在实验模板、状态和可追踪正文；OSF components 启发课题按阶段组织；Obsidian 的链接/反链说明笔记价值在上下文关系；RDM/FAIR 资料强调数据与结果的可追溯、复现和复用。综合这些公开工具和研究生实际流程，最高频需求不是更多后台字段，而是文献、实验、结果、导师沟通和写作沉淀五条路径能顺手衔接。
- 本轮计划：首页新增“调研校准 / 需求罗盘”，把五个高需求入口直接展示为可点击行动卡，让用户一打开就知道平台服务哪些真实场景，而不是面对模块列表。
- 功能边界：不新增数据库表、不新增配置项、不复制 Zotero/eLabFTW/OSF/Obsidian 的完整系统；继续复用现有文献、实验、结果、笔记、任务和周报草稿能力。
- UI 调整：新增 `need-compass` 视觉区，使用低饱和光斑、半透明纸面和轻微悬浮动效；空工作台和已有数据工作台都展示同一套需求罗盘，降低开箱迷茫。
- 产品价值：把“调研得出的高频需求”直接落到首页产品心智中，用户不用先阅读说明文档，就能从五条真实科研路径进入工作。

## 第一百三十一轮落地：会前 5 分钟导师沟通包

- 调研依据：理工科研究生和博士生日常高频沟通不是复杂审批流，而是临时见导师、组会前、周报前快速讲清“做了什么、证据在哪、下周做什么、需要导师判断什么”。Zotero/eLabFTW/RDM/Obsidian 分别解决材料源头和记录，工作台应负责会前压缩和串联。
- 本轮计划：事务页新增“会前 5 分钟”沟通包，把近期高优先级任务、进行中/失败实验、可写入结果、待读文献和待处理事务自动整理成可点击线索。
- 功能边界：不新增数据库表、不做汇报审批、不做复杂模板配置；继续复用现有周报/今日沟通单生成、导师反馈回填、任务/实验/结果/文献/事务数据。
- UI 调整：新增 `advisor-pack` 视觉区，用准备度 4 步、一组可汇报线索和 3 个会前问题替代原来的单条周报提示条；按钮优先级调整为“今日沟通单”优先，“周报草稿”和“回填反馈”作为次级动作。
- 产品价值：用户在会前不需要分别翻项目、实验、成果、文献和事务页，先在事务页得到一页可讲内容，再进入笔记页编辑草稿。

## 第一百三十二轮落地：写作材料板

- 调研依据：Zotero 负责文献源头，ELN 负责实验复盘，RDM/FAIR 强调结果证据可追溯，Obsidian 证明链接和反链有利于写作沉淀；但研究生写周报、组会和论文前最痛的是材料散落在阅读、实验、结果、导师反馈和草稿里。
- 本轮计划：笔记页新增“写作材料板”，按阅读材料、实验复盘、结果证据、组会反馈和写作草稿五类汇总数量，并提供直接筛选入口和本周素材包入口。
- 功能边界：不新增数据库表、不引入复杂编辑器、不复制 Obsidian 知识图谱；继续复用现有笔记分类、标签、正文关键词、双链和写作素材包能力。
- UI 调整：新增 `writing-board` 视觉区，使用低饱和暖纸层、轻量数字信号和来源卡片，放在笔记页 hero 与主编辑区之间，避免把材料入口藏在侧栏。
- 产品价值：写论文、周报或组会前不用分别翻文献、实验、成果和事务页，先在笔记页得到一张材料地图，再继续右侧沉浸写作。

## 第一百三十三轮落地：文献综述合成台

- 调研依据：Zotero Web API 和 Zotero 的集合/标签能力适合继续承担文献源头；Obsidian 的内部链接说明笔记价值在可连接的知识网络；OSF components 强调把研究按阶段、材料类型拆开；ELN 工具强调实验记录和模板。对研究生而言，文献页最该补的不是第二套文献库，而是从 Zotero 同步后的长列表快速进入“读后抓手”和“综述矩阵”。
- 本轮计划：文献页新增“综述合成台”，在同步/阅读栈后显示当前筛选范围、可对比文献、待收口文献和已有上下文数量，并把“生成阅读收口清单”和“生成综述矩阵”放成两步动作。
- 功能边界：不新增数据库表、不做自动综述、不自动编造文献事实；矩阵仍只是可编辑 Markdown，用户人工补问题、方法、数据、指标和可复用点。
- UI 调整：新增 `literature-synthesis` 视觉区，延续低疲劳暖纸和鼠尾草色系，用小型信号卡替代普通工具条，避免用户在文献长列表顶部找按钮。
- 产品价值：写 related work、开题综述或组会文献部分前，不需要从一长串文献里硬挑材料；先补读后抓手，再生成矩阵，减少从阅读到写作的切换成本。

## 第一百三十四轮落地：实验复现与收口板

- 调研依据：eLabFTW 这类 ELN 的核心价值在实验条目、模板、状态和正文记录；RDM/FAIR 资料强调结果要可追溯、可复现。理工科研究生实验页最需要的不是参数仓库，而是快速判断哪些实验还在观察、哪些失败要复盘、哪些完成但缺结果证据、哪些已有证据需要回填正文。
- 本轮计划：实验页新增“复现与收口板”，把当前筛选范围中的继续观察、失败复盘、缺结果证据、已有证据四类信号直接展示，并保留三项实验收口清单入口。
- 功能边界：不新增数据库表、不做完整 ELN、不做样品/仪器/权限管理；继续复用现有实验状态、结果关联、复盘任务、复盘笔记和结果证据能力。
- UI 调整：新增 `experiment-repro` 视觉区，延续低疲劳纸面风格，把复现状态信号放在 hero 与实验列表之间，减少用户在长实验列表里找未闭环条目的时间。
- 产品价值：实验页从“记录列表”进一步变成“实验闭环台”，用户能先判断今天该补观察、复盘失败、登记结果还是回填证据，再进入具体实验卡片。

## 第一百三十五轮落地：图表证据包

- 调研依据：RDM/FAIR 强调数据与结果要可追溯、可复现、可复用；科研图表写作真正需要的是路径、核心指标、复现状态和结论口径，而不是另建完整数据平台。
- 本轮计划：成果页新增“图表证据包”，把可直接讲、缺图表路径、待补可信度、待写作标记四类信号前置，并把当前筛选结果收成图表清单。
- 功能边界：不新增数据库表、不接入文件上传、不做自动作图；继续复用现有结果、图表路径、复现状态、写作素材和证据缺口清单能力。
- UI 调整：新增 `figure-pack` 视觉区，延续低疲劳暖纸层和轻量信号卡，把图表候选与证据缺口放在成果页 hero 后，减少用户从长结果列表里找图表素材的时间。
- 产品价值：组会、周报或论文写作前，用户能先看到哪些结果已经能讲，哪些还缺路径或复现，再一键生成图表清单或补证据任务。

## 第一百三十六轮落地：课题交付板

- 调研依据：OSF Projects 用 components、wiki、文件和链接组织研究阶段；eLabFTW 证明模板和状态能降低实验记录成本；FAIR/RDM 资料强调研究产出要能追溯和复用。对个人研究生而言，课题页不需要复制完整 OSF 或项目管理软件，而是要把复杂课题压成“可验收阶段 + 下一步动作 + 可转实验”。
- 本轮计划：课题页新增“课题交付板”，前置活跃课题、临近阶段、缺行动阶段和已验收阶段四类信号，并把最该处理的 3 个阶段组成交付队列。
- 功能边界：不新增数据库表、不做甘特图、不做多人协作和权限；继续复用 Project、Milestone、Task、转实验、推进笔记和筛选能力。
- UI 调整：新增 `project-delivery` 视觉区，延续低疲劳暖纸层和轻量信号卡，把阶段验收逻辑放在 hero 后、任务看板前，避免用户先陷入三列任务后台。
- 产品价值：打开课题页后先判断哪个阶段该验收、哪个阶段缺动作、哪个动作能转实验，再进入具体任务；课题管理从“维护结构”转成“推动交付”。

## 第一百三十七轮落地：科研日节奏板

- 调研依据：Zotero、ELN、OSF、RDM/FAIR 和 Obsidian 分别覆盖文献、实验、课题、数据证据和知识沉淀，但研究生每天最缺的是“现在该做哪一类动作”的节奏判断，而不是更多模块入口。
- 本轮计划：首页新增“科研日节奏板”，把一天压成上午定主线、中午补输入、下午做验证、晚上轻收口四段，并自动高亮当前时段。
- 功能边界：不新增数据库表、不做日历、不做复杂提醒；继续复用今日计划、文献队列、实验记录、结果证据、收口提醒和笔记入口。
- UI 调整：新增 `research-day-rhythm` 视觉区，使用低疲劳暖纸层、当前时段高亮和四段行动卡，放在自动整理区与需求罗盘之间，减少用户每天打开首页后的模块选择成本。
- 产品价值：忙碌用户不必先思考该进文献、项目、实验还是成果页，而是按当前时段进入最合适的研究动作，把平台从“模块集合”进一步变成“日常节奏助手”。

## 第一百三十八轮落地：设置页连接面板

- 调研依据：研究生日常使用中经常变动的是 AI Key/Base URL、Zotero 同步范围和访问密码；数据库、端口、加密密钥属于部署阶段信息，不应长期占据普通用户视野。
- 本轮计划：设置页新增“常用连接”面板，用 AI、Zotero、访问密码三张入口卡替代默认铺满的三大表单；具体表单改成可展开详情区。
- 功能边界：不新增数据库表、不改变配置存储、不把部署项搬到普通表单；继续复用现有 AppSetting、测试连接、保存配置和高级部署信息折叠区。
- UI 调整：新增 `settings-control-panel`、`settings-shortcut-card`、`settings-detail-card` 样式，让设置页从配置墙变成低压力连接中心。
- 产品价值：用户只在需要修改某个连接时展开对应表单，日常打开设置页先看到状态和入口，减少“系统后台感”和配置疲劳。

## 第一百三十九轮落地：全局快速捕捉减负

- 调研依据：繁忙研究生日常记录通常发生在切换任务、见导师、做实验或准备组会的间隙；真正省时间的入口应像收件箱一样先接住一句话，再自动放到任务、实验、成果、文献、事务或笔记，而不是要求用户先判断模块和填写表单。
- 本轮计划：重构顶部快速捕捉提示和示例，加入自动归档关键词规则，并提供轻量类型按钮用于手动覆盖去向。
- 功能边界：不新增数据库表、不引入 AI 分类、不增加配置项；仍使用现有 `quickCapture` server action 和现有模块数据模型。
- UI 调整：输入框文案从“模块枚举”改成自然语言收件箱；去向提示改为“收进：某模块”；示例减少到真实科研短句；类型 chip 展示任务、实验、成果、文献、事务、笔记六类并高亮当前判断。
- 产品价值：用户不用先想“这句话该放哪个页面”，直接输入“周五组会准备图表”“补一组对照实验”“消融提升 2%”即可落到对应工作区，降低平台使用的启动阻力。

## 第一百四十轮落地：会后反馈闭环板

- 调研依据：Zotero 的 collection/tag/API 适合保留为文献源头，eLabFTW 的实验模板和状态说明实验记录要围绕“可复用模板 + 状态收口”，OSF 的 components/wiki/files 强调课题阶段和材料可追溯，Obsidian 的链接/反链说明笔记价值在上下文关系。对研究生而言，组会和导师沟通后的最高频痛点不是会议纪要本身，而是“导师说了什么、哪些要拆成任务、哪些要回填实验/结果/文献”。
- 本轮计划：事务页新增“会后反馈闭环”板，读取最近组会/导师反馈笔记，统计未同步的 checklist，并提供一键拆成项目任务的入口。
- 功能边界：不新增数据库表、不做多人会议系统、不引入新的审批/日历/配置；继续复用现有组会笔记、`createMeetingFeedbackNote`、笔记 checklist 拆任务能力和项目页待办。
- UI 调整：在“会前 5 分钟”后增加低疲劳纸面闭环区，分为“先写原话、再拆动作、回填实验、补证据/文献”四步；右侧展示最近反馈记录和“拆成任务/查看待办”入口。
- 产品价值：用户会后不用再想反馈应该放哪里，也不用在笔记和项目之间手工搬运；导师反馈可以快速变成课题页待办，并提示后续回填实验、成果和文献。

## 第一百四十一轮落地：文献到实验桥

- 调研依据：Zotero 的官方定位是文献条目、集合、标签和 API 同步源；eLabFTW 的价值在实验条目、模板和状态；Obsidian 的价值在笔记链接和上下文。研究生读文献的高频目标不是维护第二套文献库，而是把“方法、代码、数据、对照、指标”快速转成可验证动作。
- 本轮计划：文献页新增“文献到实验桥”，自动从当前筛选文献中挑出可能含方法/代码/数据/对照/指标的候选，并前置“转实验、待验证、笔记”三个动作。
- 功能边界：不新增表、不做自动事实判断、不替代 Zotero 标签体系；继续复用现有 `createExperimentFromPaper`、`createTaskFromPaper`、`createReadingNoteFromPaper`。
- UI 调整：在综述合成台后增加低疲劳纸面桥接区，展示“读方法、转验证、留待办”三步和最多 4 篇候选文献。
- 产品价值：读到可复现方法时不再停留在文献列表，用户一键生成实验草稿或项目待验证任务，减少从阅读到实验验证的切换成本。

## 第一百四十二轮落地：实验到成果桥

- 调研依据：eLabFTW 类 ELN 的高频价值在实验条目、模板、状态和可追溯记录；RDM/FAIR 强调结果应具备元数据、复现状态和可复用证据。研究生真正痛的是实验有观察、有指标或失败现象后，没有立刻收成可讲结果，等组会/论文前再找会断线。
- 本轮计划：实验页新增“实验到成果桥”，自动挑出完成但缺结果、失败可讲、进行中有观察的实验，并前置“收证据、复盘、任务/成果”动作。
- 功能边界：不新增表、不做自动数据分析、不替代成果页；继续复用 `createResultFromExperiment`、`createExperimentReviewNote`、`createExperimentReviewTask`。
- UI 调整：在复现与收口板后增加低疲劳纸面桥接区，展示“先定观察、登记证据、回填正文”三步和最多 4 条候选实验。
- 产品价值：实验记录不再只停留在日志层，关键观察能立即流向成果页，形成组会、周报和论文可用的结果证据。
